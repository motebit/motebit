# @motebit/crypto-play-integrity — DEPRECATED

**Use [`@motebit/crypto-android-keystore`](../crypto-android-keystore/CLAUDE.md) for canonical sovereign-verifiable Android hardware attestation.**

This package is scheduled for removal at `@motebit/crypto-play-integrity@2.0.0`. It survives published only for one minor cycle of dep-resolution stability for anyone who picked it up before reading docs.

## Why deprecated

This package was scaffolded as a sibling of `@motebit/crypto-appattest` / `@motebit/crypto-tpm` / `@motebit/crypto-webauthn` — all of which verify against publicly-pinned trust anchors so that any third party can independently reach the same yes/no answer. That sibling shape is structurally wrong for Google Play Integrity:

- **Google publishes no global Play Integrity JWKS.** The verification public key is per-app, generated by Play Console for the developer, exported as `base64_of_encoded_verification_key` to `local.properties`, and never globally published. By Google's deliberate design, verification is either (a) a network round-trip to `playintegrity.googleapis.com/v1/{packageName}:decodeIntegrityToken` with the developer's service-account credentials, or (b) local decryption with a key only the developer holds.
- **Neither path satisfies the motebit invariant.** Path (a) requires a network call to Google. Path (b) requires a per-deployer secret that, by Google's contract, is not shareable with third-party verifiers.
- **The empty `GOOGLE_PLAY_INTEGRITY_JWKS = { keys: [] }`** the package shipped with at v1.0.0 was the only honest representation of "no public JWKS exists" — but the package's name, description, and place in the canonical-leaf set all imply otherwise.

The architecturally-correct Android primitive — public-anchor, third-party-verifiable, no per-deployer secrets — is **Android Hardware-Backed Keystore Attestation**. Google publishes the trust anchors at `https://github.com/android/keyattestation` (mirrored from Android's published attestation root list); device chains terminate there. That's the Android equivalent of Apple's App Attest CA model and Yubico's batch attestation root model.

The replacement package, `@motebit/crypto-android-keystore`, shipped 2026-04-26 and is the canonical path for a `platform: "android_keystore"` claim. The mobile mint cascade is being rewritten to use the on-device `KeyStore.getInstance("AndroidKeyStore")` flow with `setAttestationChallenge(SHA256(canonical body))`, replacing the `expo-play-integrity` Kotlin module.

## What this package would still be good for

Play Integrity remains a useful **operator-mode anti-fraud signal** — the verdicts (`playProtectVerdict`, `appAccessRiskVerdict`, `licensingVerdict`) carry meaningful per-deployer risk data that a relay operator might legitimately want to score. But those signals are not third-party verifiable cryptographic claims, so they do not belong in the permissive-floor crypto-leaf set alongside WebAuthn / App Attest / TPM / Android Keystore.

If a future motebit relay deployment wants Play-Integrity-as-anti-fraud, the right home is a relay-side service or an explicitly non-canonical operator-tier package. Building that speculatively today is the kind of "build-for-hypothetical-future" anti-pattern motebit's "build for endgame, not for MVPs" doctrine cuts against — wait until a real operator surfaces the requirement, then build it in the correct architectural category.

## Migration

| Before                                                                   | After                                                                                                                                |
| ------------------------------------------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------ |
| `import { playIntegrityVerifier } from "@motebit/crypto-play-integrity"` | `import { androidKeystoreVerifier } from "@motebit/crypto-android-keystore"`                                                         |
| `playIntegrityVerifier({ expectedPackageName: "com.motebit.mobile" })`   | `androidKeystoreVerifier({ expectedAttestationApplicationId })` (raw bytes captured at registration time — see new package's README) |
| `verify(credential, { hardwareAttestation: { playIntegrity: ... } })`    | `verify(credential, { hardwareAttestation: { androidKeystore: ... } })`                                                              |
| `claim.platform === "play_integrity"`                                    | `claim.platform === "android_keystore"` (new platform discriminator added to `HardwareAttestationClaim`'s union)                     |

See the new package's [CLAUDE.md](../crypto-android-keystore/CLAUDE.md) and [README.md](../crypto-android-keystore/README.md) for the canonical wiring; see [`docs/doctrine/hardware-attestation.md`](../../docs/doctrine/hardware-attestation.md) for the architectural categorization that drove this change.
