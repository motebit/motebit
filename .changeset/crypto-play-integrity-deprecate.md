Deprecate `@motebit/crypto-play-integrity` as a sovereign-verifiable hardware-attestation leaf. Use [`@motebit/crypto-android-keystore`](https://www.npmjs.com/package/@motebit/crypto-android-keystore) for canonical Android hardware attestation. Removal scheduled for `@motebit/crypto-play-integrity@2.0.0`.

## Why

This package was scaffolded as a sibling of `crypto-appattest` / `crypto-tpm` / `crypto-webauthn` — all of which verify against publicly-pinned trust anchors so any third party reaches the same yes/no answer. That sibling shape is structurally wrong for Google Play Integrity:

- Google publishes no global Play Integrity JWKS. The verification public key is per-app, generated by Play Console for the developer, and never globally published — by Google's deliberate design.
- Verification is either a network round-trip to `playintegrity.googleapis.com` (with the developer's service-account credentials) or local decryption with a per-app private key. Neither path is third-party verifiable.
- The empty `GOOGLE_PLAY_INTEGRITY_JWKS = { keys: [] }` shipped at v1.0.0 was the only honest representation of "no public JWKS exists" — but the package's name, description, and place in the canonical-leaf set all implied otherwise. That category error is what this deprecation corrects.

The architecturally-correct Android primitive — public-anchor, third-party-verifiable, no per-deployer secrets — is **Android Hardware-Backed Keystore Attestation**, shipped at `@motebit/crypto-android-keystore` since 2026-04-26. Device attestation chains terminate at Google's published Hardware Attestation roots (RSA + ECDSA P-384), the Android equivalent of Apple's App Attest CA model.

## What changed

- All public exports of `@motebit/crypto-play-integrity` carry `@deprecated` JSDoc tags pointing at `@motebit/crypto-android-keystore`.
- `package.json` description, README, and CLAUDE.md lead with the deprecation + the architectural reason.
- The verifier and tests still function; this is a category correction at the doctrine + naming level, not a code-level removal.

## Future scope (intentionally not in this package)

Play Integrity remains a useful **operator-mode anti-fraud signal** — Google's `playProtectVerdict`, `appAccessRiskVerdict`, and `licensingVerdict` carry meaningful per-deployer risk data. But those signals are not third-party verifiable cryptographic claims, so they don't belong in the permissive-floor crypto-leaf set. If a future motebit deployment wants Play-Integrity-as-anti-fraud, it lands as a relay-side service or an explicitly non-canonical operator-tier package — not here. See `docs/doctrine/hardware-attestation.md` § "Three architectural categories".

## Migration

```text
import { playIntegrityVerifier } from "@motebit/crypto-play-integrity";
// →
import { androidKeystoreVerifier } from "@motebit/crypto-android-keystore";

// claim.platform: "play_integrity"
// →
// claim.platform: "android_keystore"

// verifiers: { playIntegrity: ... }
// →
// verifiers: { androidKeystore: ... }
```
