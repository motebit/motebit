**Four-part-triple drift gate — graduates `check-typed-truth-perception` from 3-of-3 to 5-of-5 coverage for narration-class typed-truth fields.** Closes the producer-gap bug class witnessed 2026-05-12 in commit `8b1d6605`, where wire + prompt + validator shipped for `task_step_narration` with no producer; the validator's pass-through fired 100% of the time, the triple was structurally complete on the registry side and inert at runtime. Caught by user intuition before push, fixed in `cf1227d0` by adding `extractNarrationTag`; the doctrine memo promoted the four-part naming the same day. This gate makes that producer-gap pattern structurally impossible to reland.

**Half-4 + Half-5 — explicit producer + validator sources for narration class.** `TypedTruthField` gains two optional fields: `producerSources` (the runtime code that populates the field at emission time — for narration, the tag parser like `extractNarrationTag`) and `validatorSources` (the runtime code that checks the field against wire-level typed truth before consumption — for narration, validators like `validateTaskStepNarration`). Both REQUIRED for `dishonesty-narration` class entries; ignored for other classes (tool-result classes have implicit producer = dispatch which Half-2 covers; dishonesty-persistent has implicit validator = `DISHONESTY_RULES` which Half-3 covers). Half-2 (dispatch presence) is skipped for narration class — narration doesn't flow through a wire-level dispatch surface, so requiring a dispatch source would either conflate dispatch/producer (the prior state) or fail the gate spuriously.

**Why only narration class.** Tool-result classes already have producer-coverage implicitly: the action-executor's dispatch IS the producer (it emits the field directly on the result struct). Dishonesty-persistent classes already have validator-coverage implicitly: every entry MUST appear in `DISHONESTY_RULES` (Half-3 enforces). Narration is the class where BOTH producer + validator are explicit separate code paths (model text → tag parser → typed field → validator → consumer); the registry must name both, and missing either is the half-shipped pattern. The other classes (transient / affordance / positive-signal / control-state / transparency) are not validation-applicable by design — `transient` has no walk-back semantics, `affordance` is a hint not a failure signal, `positive-signal` flags success the model SHOULD claim, `control-state` is about authority not truth, `transparency` is logging.

**The grade is structural source-presence.** The gate forces the registry to NAME a producer + validator file for every narration-class field; if you can't name one, you don't have one. End-to-end data-flow tracing ("the producer's return value actually populates the field at runtime") remains a reviewer-discipline check on top — but the half-shipped registry pattern is closed mechanically. Doctrine cross-cuts: `docs/doctrine/runtime-invariants-over-prompt-rules.md` § "The four-part typed-truth structure" updated to mark the deferred reviewer-discipline note as shipped; `docs/drift-defenses.md` #80 entry rewritten to name Half-4 + Half-5.

**Adversarially tested before landing.** Synthetic break of `producerSources: []` on `task_step_narration` produces a Half-4 violation with the actionable remediation pointing at `extractNarrationTag` and the doctrine. Synthetic break pointing `producerSources` at the wrong source file (`dishonest-closing.ts` instead of `core.ts`) produces a "declared but field not found" violation. Same shape for `validatorSources`. The probe surface confirms each half fires independently.

13 typed-truth fields registered, all four-part-complete; 1 narration-class field with explicit producer + validator (currently only `task_step_narration` — when a second narration field ships, a `NARRATION_RULES` table emerges to parallel `DISHONESTY_RULES`, paralleling Half-3's append-only-enforcement shape).
