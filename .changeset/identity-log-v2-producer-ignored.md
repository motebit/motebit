Flip the identity-transparency log producer to `merkle-sha256-rfc6962-v2` — PR4 of the RFC 6962 §2.1 domain-separation migration, the third real v2 producer (doctrine: `docs/doctrine/merkle-tree-hash-versioning.md` §8). Producer side only: the portable verifier (`@motebit/crypto`'s `verifyIdentityBindingAnchored` + `IdentityLogInclusionProof.tree_hash_version?`) accepted v2 from PR1 part 2b (deploy-verifier-first ordering, threat-model §4(d)), so no crypto change.

`services/relay/src/identity-log.ts`: `buildIdentityLog(bindings, treeHashVersion?)` threads the version into `identityLogLeaf` (the `0x00` leaf tag — already in `check-merkle-tree-hash-canonical`'s `LEAF_BUILDERS`) and `buildMerkleTree` (the `0x01` node tag); the served `IdentityLogProof` stamps `tree_hash_version` for v2 and omits it for v1 (absent ⇒ v1). `services/relay/src/identity-log-anchoring.ts`: module constant `IDENTITY_LOG_TREE_HASH_VERSION` emits v2; the root-change trigger in `runIdentityLogAnchorTick` recomputes under the same v2 so a v1 stored root doesn't read as a phantom change and re-anchor every tick (a one-time re-anchor on the v1→v2 transition is intended). The `/identity` proof endpoint (`identity-transparency.ts`) rebuilds each snapshot under its stored version so a verifier's reconstructed root matches the on-chain commitment.

Per-anchor version persistence is the load-bearing correctness detail: a new `relay_identity_log_anchors.tree_hash_version` column records the version each snapshot was hashed under. The column lands in BOTH the startup `createIdentityLogAnchorTables` `CREATE TABLE` (fresh DBs) and migration v28's PRAGMA-guarded `ALTER` (existing prod DBs) — with the INVERSE ordering of PR3's v27: `createIdentityLogAnchorTables` runs AFTER migrations, so on a fresh DB the v28 `cols.length > 0` guard SKIPS and the ALTER fires only on existing DBs. Already-anchored pre-PR4 snapshots read NULL ⇒ `merkle-sha256-plain-v1`, so their on-chain-committed v1 roots still reconstruct under v1; the served proof omits the field for legacy v1 (the legacy id is never re-emitted). The signed root payload is unchanged — the version binds transitively through `merkle_root`.

No spec change and no Option-A frontmatter: unlike agent-settlement (§3 SCITT/RFC 6962 citation) and credential-anchor, no spec makes an RFC-6962 claim about the identity-transparency log, and its inclusion-proof endpoint (`GET /api/v1/identity/:motebitId`) is `@experimental` with no wire-format spec section. So there is no spec-vs-code drift to close here — the v2 flip is pure forward domain separation; the Option-A declaration lands when that endpoint graduates to a versioned spec. PR4's gate coverage is assertion (2), the leaf-builder route-through. Tests: the bundle→`verifyIdentityBindingAnchored` round-trip now runs v2 end-to-end (the served proof carries `tree_hash_version: "merkle-sha256-rfc6962-v2"`, the anchored root differs from a v1 build), plus a legacy v1 anchor (NULL column) verifies with the field absent.
