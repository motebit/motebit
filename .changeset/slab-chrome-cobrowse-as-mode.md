**Third slice of PR 1 of the agent-surface pivot — cobrowse-as-mode reshape.** Closes the polarity-asymmetry on the user-state side: the doctrine renames the cobrowser-shaped chrome from "the persistent default" to "an entered mode," but until this slice the user-state chrome carried no explicit indicator that it was a MODE, and there was no protocol-level path for the user to YIELD drive back to motebit (only motebit-side `releaseControl` existed). Both halves close here: the user-state chrome surfaces a "motebit waiting · hand back" chip that is both ambient mode-indicator AND handoff-back affordance; the protocol gets a symmetric `yield_control` transition that the surface routes to. Doctrine: [`chrome-as-state-render.md`](https://github.com/hakimlabs/motebit/blob/main/docs/doctrine/chrome-as-state-render.md) § "user register — cobrowse mode entered."

**Protocol — `yield_control` (additive, `@alpha`).** New entry on the `CO_BROWSE_TRANSITION_KINDS` closed registry, with a `yieldControl(by: "user")` method on `CoBrowseControlMachine` that goes from `user` → `motebit` with no approval needed (user's identity is the trust root; the user unilaterally decides who drives, including handing back). Symmetric protocol-level partner to `releaseControl` (motebit-yields) on the user side. Why a distinct transition rather than re-using `requestControl + auto-grant`: a single explicit `yield_control` produces one audit event with the right semantics ("user handed back"); the request-then-grant compound would produce two events that a verifier would have to recognize as a paired pattern. The audit log is the source of truth for "who was driving when"; one transition kind keeps the log legible.

**The surface affordance — chip-button, single mechanism.** The user-state chrome now includes a `"motebit waiting"` chip in the trail (after the ←/→/↻ history buttons). Promoted from a static label to a `<button type="button">` so click semantics, focus-ring, and keyboard activation come from the platform; `aria-label="Hand back to motebit — motebit is watching and will resume driving"` carries the affordance's full intent for assistive tech. Calm-default visual treatment (borderless tinted text, transparent background, hover lifts opacity) inherits the chrome's history-button family. Click dispatches `motebit:cobrowse-back` — the same CustomEvent the `/back` slash command uses. Web-app handler calls `machine.yieldControl("user")` directly. Surface-determinism preserved: typed CustomEvent → typed-capability call, no AI-loop routing; `check-affordance-routing` approves by construction.

**The full PR 1 lifecycle now reachable from any surface.** Motebit-default register renders; user invokes `/wheel` or taps the URL chip → state flips to user + URL bar focuses (the prior slice's gotcha closure); user navigates / clicks / types in the page; user invokes `/back` or taps the "motebit waiting" chip → state flips back, motebit resumes the narration register naturally. Single mode-flip mechanism in each direction; the slash command and the chip are surface affordances that converge on the same wire event. Future polish (gestures, voice, explicit Take-the-Wheel button) compose against the same events.
