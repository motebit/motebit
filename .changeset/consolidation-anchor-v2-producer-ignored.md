Flip the consolidation-anchor producer to `merkle-sha256-rfc6962-v2` — PR5 of the RFC 6962 §2.1 domain-separation migration, the fourth real v2 producer and the FIRST runtime-side one (PR2–PR4 were all relay; doctrine: `docs/doctrine/merkle-tree-hash-versioning.md` §8). Producer side only: the portable verifier (`@motebit/encryption`'s `verifyConsolidationAnchor` + `ConsolidationAnchor.tree_hash_version?`) accepted v2 from PR1 part 2b (deploy-verifier-first ordering, threat-model §4(d)), so no crypto/encryption/wire-schemas/spec-schema change.

`packages/runtime/src/motebit-runtime.ts`: `anchorPendingConsolidationReceipts` hashes each signed receipt via `canonicalLeaf(r, v2)` (the `0x00` leaf tag) instead of the prior `canonicalSha256(r)`, and builds the tree via `buildMerkleTree(leaves, v2)` (the `0x01` node tag); module constant `CONSOLIDATION_TREE_HASH_VERSION` emits v2.

**Persistence is the key shape difference from PR2/PR3/PR4 — no migration, no DB column.** The `ConsolidationAnchor` object IS the persisted record: the runtime emits it into its event log as a `ConsolidationReceiptsAnchored` event, and the object already carries the optional `tree_hash_version?` field (PR1 part 2b). So "persist the version per root" = set the field on the emitted anchor object. Backward-compat is automatic: anchors emitted before this flip have no field ⇒ absent ⇒ `merkle-sha256-plain-v1`, and `verifyConsolidationAnchor` resolves each anchor's own field standalone — there is no proof-serve-time reconstruction from a stored column (unlike the relay batch anchors), and dedup is by `receipt_id`, not by root, so no re-anchor-loop concern. The signed receipts and their roots are unchanged for legacy anchors; the version binds transitively through `merkle_root`.

**Gate touches TWO arms** (unlike PR4's zero), both probed: the runtime producer `anchorPendingConsolidationReceipts` joins `check-merkle-tree-hash-canonical`'s `LEAF_BUILDERS` (assertion 2, leaf-builder route-through), and `consolidation-receipt-v1.md` §4.6 + Option-A frontmatter (`tree_hash_producer: packages/runtime/src/motebit-runtime.ts`) activates a third assertion-4 spec-claim → producer declaration. Unlike PR4's `@experimental` identity-log endpoint, consolidation-receipt is a real Stable wire spec (§4.1 leaf / §4.3 Merkle / §4.5 verification), so the Option-A declaration is appropriate — it cited no RFC 6962 contract before, so this is a proactive consistency add, not a drift fix.

Tests (`packages/runtime/src/__tests__/proactive-wire.test.ts`): the batch→recompute round-trip now runs v2 end-to-end (the anchor carries `tree_hash_version: "merkle-sha256-rfc6962-v2"`, the recomputed root uses `canonicalLeaf(r, v2)` and differs from the v1 `canonicalSha256(r)` build — proving the `0x00` tag applied); a `verifyConsolidationAnchor` round-trip over a ≥2-receipt set (exercising the `0x01` node tag); and a legacy anchor with the field stripped verifies (absent ⇒ v1).
