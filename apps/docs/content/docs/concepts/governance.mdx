Governance is the surface tension. It controls what crosses the boundary between the agent's interior (memory, tools, state) and the outside world. Without it, a tool-using agent is a liability.

## PolicyGate

Every tool call passes through the `PolicyGate` before execution:

```ts
import { PolicyGate } from "@motebit/policy";

const gate = new PolicyGate({
  operatorMode: false,
  requireApproval: ["writeFile", "shellExec"],
  budgets: { maxCallsPerWindow: 20, windowMs: 60000 },
  allowList: ["readFile", "webSearch", "recallMemories"],
  denyList: [],
});
```

The gate evaluates each tool call against:

1. **Allow/deny lists** — Explicit overrides
2. **Risk classification** — Tools are classified by risk level (R0-R4)
3. **Operator mode** — High-risk tools (R2+) require operator mode to be enabled
4. **Budget enforcement** — Rate limits and time budgets per turn
5. **Approval requirements** — Specific tools can require explicit user approval

## Risk levels

Every tool gets a risk classification:

| Level | Description | Examples |
|---|---|---|
| `R0_READ` | Read-only, no side effects | `readFile`, `webSearch`, `recallMemories` |
| `R1_ANNOTATE` | Creates metadata, no external effect | `listEvents`, `readUrl` |
| `R2_WRITE` | Writes to local storage | `writeFile` |
| `R3_EXECUTE` | Executes code or commands | `shellExec` |
| `R4_MONEY` | Financial transactions | Payment tools |

The `classifyTool` function infers risk from tool names and descriptions. MCP tools from untrusted servers default to requiring approval.

## Operator mode

Operator mode is a PIN-protected escalation that unlocks high-risk tools:

1. User enables operator mode in settings
2. System prompts for a 4-6 digit PIN
3. PIN is hashed with SHA-256 and stored in the OS keyring
4. While active, R2+ tools can execute (still subject to approval requirements)
5. Disabling operator mode re-locks high-risk tools immediately

```ts
// Enable operator mode
const result = await runtime.setOperatorMode(true, "1234");
if (!result.success && result.needsSetup) {
  // First time — need to create PIN
  await runtime.setupOperatorPin("1234");
  await runtime.setOperatorMode(true, "1234");
}
```

## Audit trail

Every policy decision is logged:

```ts
interface ToolAuditEntry {
  turnId: string;
  callId: string;
  tool: string;
  args: Record<string, unknown>;
  decision: "allowed" | "denied" | "requires_approval";
  result?: string;
  timestamp: number;
}
```

The audit log is append-only and immutable. It records what was attempted, what was allowed, what was denied, and the result of every tool execution.

## MemoryGovernor

The `MemoryGovernor` applies governance to memory formation:

- Memories above a sensitivity threshold can be rejected entirely
- Retention rules are enforced per sensitivity level
- The governor checks every memory candidate before it enters the graph

```ts
const governor = new MemoryGovernor({
  retentionRules: {
    [SensitivityLevel.None]: Infinity,
    [SensitivityLevel.Personal]: 365 * 24 * 60 * 60 * 1000,
    [SensitivityLevel.Medical]: 90 * 24 * 60 * 60 * 1000,
    [SensitivityLevel.Financial]: 90 * 24 * 60 * 60 * 1000,
    [SensitivityLevel.Secret]: 30 * 24 * 60 * 60 * 1000,
  },
  rejectSensitivity: SensitivityLevel.Secret,
});
```

## Content sanitization

The `ContentSanitizer` defends against prompt injection in tool results. When a tool returns data from an external source, the sanitizer checks for LLM prompt injection attempts and flags suspicious content:

```ts
// In the streaming output
{ type: "injection_warning", tool_name: "webSearch" }
```

MCP tool results are wrapped in `[EXTERNAL_DATA]` markers to clearly delineate the trust boundary between internal and external content.

## The principle

Fail closed. If the policy engine errors, the tool call is denied. If sensitivity classification fails, the memory is rejected. If the keyring is unavailable, operator mode can't be enabled. Every failure mode defaults to the more restrictive option.
