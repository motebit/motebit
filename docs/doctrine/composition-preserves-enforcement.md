# Composition preserves enforcement

> **Motebit must prove not only that its components are correct, but that its intended guarantees remain enforced when those components are composed into the deployed system.**

This is the obligation the gate suite does not individually carry. Every gate proves a _local_ property of a part; none of them proves that the part is actually reached, with the right inputs, in the artifact that ships. The gap between those two is where a month of real defects lived — so the obligation gets named, and its enforcement mechanism gets built, rather than being rediscovered one production incident at a time.

The two load-bearing words are deliberate. **"Remain enforced"** — not "survive": a guarantee can still _exist_ in the source and never be _acted on_ by the running system. **"Deployed system"** — not "running under test": a composition that CI assembles correctly can still be assembled wrong in the shipped image (different config resolution, unwired keys, a dropped call). The sentence fails exactly the defects below, which is what makes it a test and not a slogan.

## The flaw class, with a birthdate

Trace the commit history for the family whose own messages admit a capability existed but did not fire — `inert / dormant / shadowed / silent / lost`. It does not scatter. It clusters, and the first cluster has a date:

- **Mid-June 2026** — the CLI-entry sev-1s: [`motebit-verify`](../../packages/verify) inert for every installed invocation (symlink'd-bin entry guard), the relay key silently dead via its bin, cloud-usage debits silently dropped. A correct binary, an unreached entry point.
- **Late July 2026** — the routing-transcript / federation arc: the transcript producer dormant in deployed molecules because delegator signing keys were unwired (`#357`); the minted transcript lost at egress in branch churn (`#358`); the `#188` discover-signature sunset **inert in production because `server.ts` shadowed the constant** (`#359`); and the settlement-authority payer seam, whose first version enforced only at the relay — _after_ the payer had already broadcast — and so protected records but not funds ([`settlement-authority-binding`](settlement-authority-binding.md)).

The flaw began mid-June because that is when the system crossed a complexity threshold and the dominant bug **changed shape**: from _logic-wrong_ (a part computes the wrong thing; a unit test catches it) to _enforcement-not-local_ (the part is correct and green, but the deployed path does not reach it). The root cause is structural, not a discipline lapse: in a 53-package DAG a feature's **definition** — type, crypto law, conformance vector, all passing — and its **activation** — the deployed process invoking it with resolved config and wired keys — live in _different packages_. Every gate we had proved definition. Nothing proved activation.

## Honest calibration — what the gates do and don't prove

Do not overstate the starting position. Our gates predominantly prove **local** properties — dependency-layer membership ([`check-deps`](../../scripts/check-deps.ts)), generated-file consistency, closed-registry canonicality, declaration presence, formatting, wire-format coverage. A few reach further ([`check-gates-effective`](../../scripts/check-gates-effective.ts) perturbs each gate to prove it still fires; [`check-money-authority`](../../scripts/check-money-authority.ts) enforces the R4 standing-authority composition). But **none individually proves that the complete deployed path preserves every intended guarantee.** That framing matters: it makes the gap _structural_ — a missing category of proof — not a coverage hole to be closed by raising a threshold.

Symmetrically, do not overstate the fix. A harness earns the claim "proves enforcement in the deployed system" only if it builds the **actual production artifact**, boots it through the real entry point, resolves production-equivalent configuration, uses real credential and key wiring, crosses the persistence and process boundaries, exercises the external interfaces, verifies forbidden side effects do not occur — **and goes red when each critical connection is deliberately severed.** A harness that composes the parts in-process is just another _modeled_ composition, which is precisely the thing that passed for `#357`. The severing test is the whole point: an activation check that does not fail when you re-shadow the constant proves nothing.

## The response is two-part, and detection is the weaker half

The first response already shipped: the effective-config harness ([`#360`](../../services/relay/src/relay-config.ts), `buildRelayConfigFromEnv` + `SECURITY_BOUNDARY_DEFAULTS` + `probeSecurityBoundaries`) — it proves the _resolved_ security-boundary config at the enforcement point, not the exported constant, and [`check-security-default-wiring`](../../scripts/check-security-default-wiring.ts) forbids a hard-coded literal shadowing a canonical default. That is real, and it is the **detection** half.

But detection is the weaker half, and naming it as the whole answer would be a mistake. _A test detects a shadowed constant; a better configuration architecture makes shadowing structurally impossible._ This is [`runtime-invariants-over-prompt-rules`](runtime-invariants-over-prompt-rules.md) — make illegal states unrepresentable — applied to composition: one resolution path, no second literal that _can_ diverge, so the shadow does not compile rather than getting caught after the fact. So the doctrine is a pair, ranked:

1. **Reduce the seams where enforcement can disappear.** Prefer a structure where the guarantee cannot be composed away — a single resolution path, a producer that cannot be constructed without its key, an egress that is the return type rather than a side assignment. Structural impossibility beats detection wherever it is affordable.
2. **Prove composition for the seams that remain.** Where a seam is irreducible, an **activation-conformance** pass exercises the deployed artifact end to end and proves each pipeline link — identity → authorization → policy → action → receipt → settlement → trust — is _live_, with the deliberate-severing test as its integrity check.

## The pattern, seeded — needle-severing vs behavioral-severing

Inc 1 establishes the mechanism on one real incident, and the distinction it draws is the whole point. Every one of `#357`/`#358`/`#359` already had a _static_ defense: a needle in a gate ([`check-routing-transcript-emission`](../../scripts/check-routing-transcript-emission.ts)'s eight chain links; [`check-security-default-wiring`](../../scripts/check-security-default-wiring.ts)'s canonical-constant reference), and [`check-gates-effective`](../../scripts/check-gates-effective.ts) already perturbs those gates to prove they fire. Call that **needle-severing**: reintroduce the failure as an absent _source string_ and prove a gate catches it. It is necessary but not sufficient — and `#358` is the proof, in its own commit message: _"the result-type field and every other link survived, so the emission gate stayed green"_ while the runtime behavior was broken. A string can be present and the deployed path still not execute it.

So the skeleton is **behavioral-severing**: reintroduce the failure as the real runtime _condition_, run the actual path, and prove the guarantee's assertion goes red. Two increments build it for `#357`, and the distinction between them is itself doctrine:

- **Inc 1 — the runtime-level negative fixture** ([`execute-granted-delegation.test.ts`](../../packages/runtime/src/__tests__/execute-granted-delegation.test.ts) → `SEVERING (reintroduces #357)`): the same paid hire as the happy-path transcript test but with `signingKeys` **omitted** — the dormancy condition (`_signingKeys` null ⇒ producer skips minting). It asserts the hire still settles while the transcript is absent from both the result seam and the buffer. Honest classification: this proves the _runtime_ behaves correctly in both key states — it is a negative-control fixture, **not** activation conformance, because it constructs `MotebitRuntime` directly, one level _below_ the composition root where `#357` actually lived. If the production builder dropped the keys again, this test would stay green.
- **Inc 2 — the composition-root activation conformance** ([`money-runtime-activation.test.ts`](../../packages/molecule-runner/src/__tests__/money-runtime-activation.test.ts)): drives the **real production builder** (`defaultCreateMoneyRuntime`) with a real keypair, the real self-grant path (`selfIssueGrant` + `mintTick`), and `runMolecule`'s real `policyOverrides` composition, through a real ranked paid hire. The positive assertion binds the minted transcript to the builder's wiring — `delegator_public_key` must equal the identity hex the builder received, and the signature must verify. The hostile case ran the defect-shaped mutation: severing the builder's `signingKeys` line (the exact `#357` diff) turns the positive test red at `toHaveLength(1)` while the hire still runs to broadcast. The positive test is the permanent guard; the severing run is the one-time proof it discriminates.

Two lessons the increments taught, kept on purpose. First, the mutation must be defect-shaped: Inc 1's perturbation (add keys to the negative fixture) only proved the fixture non-vacuous; Inc 2's (sever the production wiring, watch the _positive_ guarantee fail) reproduces the actual incident class. Second, activation tests must be **production-faithful in composition, not just in code path**: Inc 2's first draft passed `policyOverrides: {}` — a config production never runs (`runMolecule` always feeds `DEFAULT_POLICY_OVERRIDES`) — and failed `missing_scope` before selection, silently testing a different composition. The transcript-optional posture the tests encode is the decided **graded-evidence** model, not an accident: settlement proceeds without a transcript ([`routing-decision-transcript.md`](routing-decision-transcript.md) — reveals, never authorizes) while the fleet-level conformance probe WARNs on its absence — which is precisely how `#357` was detected. That is the shape Inc 3+ extends — one behavioral-severing pair per pipeline link, each paired with the seam-reduction of principle 1 wherever the dormancy can be made structurally impossible instead.

## Provisional until hostile evidence

The class is **not** closed by this document. Doctrine can name the obligation; only hostile production-path evidence can establish that it has been met. The bar is concrete: the activation-conformance pass must catch **reintroduced** versions of `#357`, `#358`, and `#359` against the real deployable — the dormant producer, the lost egress, the shadowed constant — each deliberately reinjected, each turning the pass red. Status against that bar, kept precise:

- **`#357` — met at the composition root** (Inc 2: the severed builder turns the positive test red). Sibling audit alongside it: the producer's caller surface is exactly two compositions — the molecule-runner spend handle (guarded) and the CLI `delegate.ts` subcommand (wires keys from identity); the CLI `runtime-factory` runtime carries no `signingKeys` but is **latent, not live** — it has no granted-spend caller, so the trigger to wire it is a granted-spend surface appearing on the daemon.
- **`#358` — met at the consumed seam** (Inc 3a: severing `mintedTranscript = transcript` while leaving the buffer push intact — the _exact_ incident shape, the one the pre-fix regression test stayed green through — turns the post-`#358` result-seam assertion red at `expected undefined to be 'alice-worker'`). The guard existed since the fix; what Inc 3a added is the severing run proving it discriminates.
- **`#359` — met at the booted entry** (Inc 3b: [`booted-entry-activation.test.ts`](../../services/relay/src/__tests__/booted-entry-activation.test.ts) spawns the **real deployed entry** — `server.ts` as a child process with a production-shaped strict env, real env resolution, real HTTP listener — and runs `probeSecurityBoundaries` against it from outside; every applicable boundary must report strict, and the `#359` boundary specifically must be probed, not skipped). The severing run reintroduced `#359`'s shape at today's seam — the entry shadowing the built config (`requireDiscoverSignature = false` between builder and `createSyncRelay`) — and both tests went red at `POST /federation/v1/discover → 200 (strict expects 403)`. The decisive detail: under that exact severing, the static needle gate (`check-security-default-wiring`) **stayed green** — the entry still references the canonical constant through the builder; the shadow is a post-build mutation the needle layer structurally cannot see. The booted probe is the only layer that catches it. Inc 4 added the second rung: the same test now also boots the **compiled artifact** — `node dist/server.js`, the exact command `run.sh`, `package.json#start`, and the DEPLOY.md systemd unit all exec (freshness guaranteed by turbo's `test` → `build` dependency). Its severing run applied the same shadow to `dist/server.js` **only**, source untouched — an artifact-not-source defect — and the differential came out exactly as the ladder predicts: the source rung stayed green, the compiled rung went red at `POST /federation/v1/discover → 200 (strict expects 403)`. A defect that exists only in the built output (stale or corrupted emit, build-pipeline drift) is structurally invisible to every source-level layer; the artifact rung is the only one that observes it. Both rungs are kept deliberately: a source-green/artifact-red differential localizes the fault to the build step. The rung above completes the tier: the shipped **container image**. `publish-images.yml` now boots the built image before anything is pushed — the `pnpm deploy`-flattened filesystem, the pinned base-image node, `run.sh` itself, with the image's own production env defaults and no overrides — and runs [`probe-image.mjs`](../../services/relay/scripts/probe-image.mjs) from a sibling container sharing its network namespace. The probe imports the image's OWN compiled `probeSecurityBoundaries`, so the probe bytes are the artifact's bytes; a lost boundary blocks the push, meaning a bad image is never published. Its severing run derived a severed image from the real one (the `#359` shadow patched into the image's compiled `server.js`, nothing else changed) — the pristine image probed green, the severed image probed red at `POST /federation/v1/discover → 200 (strict expects 403)`.

### The pipeline-link ladder

With the booted-artifact tier standing, the fan-out is one behavioral-severing PAIR per pipeline link (identity→authz→policy→action→receipt→settlement→trust), each paired with seam-reduction where affordable. Status, kept precise:

- **identity→authz — paired.** Seam-reduction: `mintAudienceToken` (@motebit/crypto) + `check-token-mint-canonical` ended the 23-site restatement of `iat`/`exp`/`jti`/TTL. Behavioral pair: [`booted-authz-activation.test.ts`](../../services/relay/src/__tests__/booted-authz-activation.test.ts) boots the compiled artifact, provisions a device over the real HTTP API, and drives the exact client-shaped token at the real route — accept-half (a `market:query` device token reaches market discovery: the f9d875b1 fix observed from outside; the incident hid because server tests used master while client tests mocked fetch — this suite makes the two contract halves meet in the deployed artifact) and reject-half (the same key's `sync` token refused; unauthenticated refused). Severing runs proved BOTH directions: removing the catch-all exclusion (the exact f9d875b1 shape — registered-but-rejected) turns the accept-half red at 401; removing the `dualAuth` registration turns the reject-halves red at 200.
- **policy→action — paired at the composition root** (in-process; the booted rung remains). Seam-reduction: the R4-money authority AND is single-sourced in `PolicyGate.validate` (step 8b/8c) with one audited producer of `verifiedGrant` (`verifyGrantForTurn`), statically pinned by [`check-money-authority`](../../scripts/check-money-authority.ts) ([`memory-never-confers-authority`](memory-never-confers-authority.md)). Behavioral pair: [`money-runtime-activation.test.ts`](../../packages/molecule-runner/src/__tests__/money-runtime-activation.test.ts) gains the composition-root NEGATIVES — a revoked self-grant, an out-of-scope grant, and a ceiling-tampered grant, each driven through the REAL `defaultCreateMoneyRuntime` (not the direct-`MotebitRuntime` construction the runtime-layer negatives use), each asserting fail-closed (`requires_verified_grant` / `missing_scope`) with no selection and no rail broadcast. The gap they close: every fail-closed proof lived one layer below the builder, so a drift in the builder's policy wiring (e.g. `denyAbove` or the self-grant scope) would leave them all green.
- **Discovered by the severing, not by design:** the revocation list reaches `verifyGrantForTurn` only through the caller-threaded `params.delegation.revocations ?? []` in `executeGrantedDelegation` — a fail-open default (`?? []`) that NO static needle covers. `check-money-authority` pins the verify call, the null-check, the scope fence, and the meter-wrap, but not revocation threading; severing `?? []` → `[]` turns the revoked-grant negative red while the gate stays green. This is the same needle-vs-behavioral distinction #359 drew at the entry, now inside the money path: the behavioral negative is the only layer that catches a dropped revocation thread. (Deferred-with-trigger: extend the gate's needle #4 to require the revocation argument be caller-threaded, when the money path next changes.)
- **action→receipt — booted rung shipped** (the delegator-side egress slice stays in-process). The guarantee: an action settles ONLY on a signed `ExecutionReceipt` that verifies under Ed25519. `#357` (transcript producer dormant when the composition root didn't wire signing keys) and `#358` (minted receipt never threaded onto the result) prove the DELEGATOR-side egress in-process; those cover the producer, not the relay's settlement-authorizing gate. The relay-side gate — `POST /agent/:id/task/:taskId/result` → `handleReceiptIngestion`'s `verifyExecutionReceipt` → `if (!receiptValid) return { verified: false }` (403) — was only ever exercised in-process (`money-runtime-activation.test.ts`'s header recorded the missing booted rung). Behavioral pair: [`booted-receipt-activation.test.ts`](../../services/relay/src/__tests__/booted-receipt-activation.test.ts) boots the compiled artifact, provisions a worker over the real HTTP API, queues a free self-delegation task, and POSTs receipts at the real `/result` route — accept-half (a valid worker-signed receipt bound to its task → 200) and forgery-half (a byte-tampered signature → 403). Transport auth uses the operator master token so the only variable under test is the receipt body. Severing run: `let receiptValid = await verifyExecutionReceipt(...)` → `let receiptValid = true` + rebuild dist → the forgery-half flips to 200 (`expected 200 to be 403`) while the accept-half holds. The gate goes red exactly when the artifact stops verifying the signature.
- **Discovered by the action→receipt severing, not by design:** `handleReceiptIngestion` looks up the verifying key from `receipt.motebit_id` (a receipt-body field) and, on a first-pass miss against the registered key, falls back to the key EMBEDDED in the receipt — heal-writing it onto `agent_registry.public_key` for `receipt.motebit_id`. An agent that self-delegates can mint a `task:result` token for its own task, then POST a receipt claiming a VICTIM's `motebit_id` with its own key embedded; the fallback verifies against the embedded key and OVERWRITES the victim's registered key — a cross-identity registry-key hijack no static needle covers (every wiring string is present; the missing guard is a runtime condition). The naive fix — bind `receipt.motebit_id === entry.task.motebit_id` — is WRONG and was reverted before merge: routed delegation legitimately separates the task TARGET (`task.motebit_id`) from the EXECUTOR (`receipt.motebit_id`), so binding them reds ~19 delegation/credential tests (A→B where B executes A's routed task; the internal WS/MCP ingestion sites carry the same target≠executor shape). The correct invariant is subtler and touches the key-rotation path — the heal-write must be bound to key material already associated with `receipt.motebit_id` (a registered DEVICE of that identity), never to an arbitrary self-signed key, or the overwrite-an-existing-key branch must be removed (heal only when no key is on file). Deferred-with-trigger to the embedded-key-fallback seam ([`verify-family-fail-closed`](verify-family-fail-closed.md)): this is a money/security path, so the arc records the gap rather than ship a hasty rebind. Trigger: the next change to `handleReceiptIngestion`'s key resolution, or a second consumer of the embedded-key heal.
- **receipt→settlement — booted rung shipped for the P2P (ledger-of-record) lane** (the relay-custody money-conservation gate stays in-process — see the remainder below). The prior link ends at "the receipt verifies or is refused (403)"; this link is everything AFTER `verified`. The relay is the LEDGER OF RECORD ([`endgame`](../../README.md) / [`clearing-house-not-thin-waist`](clearing-house-not-thin-waist.md) — the moat is trusted history, not custody), so on the P2P lane its ENTIRE enforcement contribution is the RECORDING: a signed `relay_settlements` audit row committing "I observed this settle P2P at `amount_settled` + `platform_fee`," which the p2p-verifier later checks against the onchain legs. Corrupt that recording and the dispute-grade history is corrupt. The P2P lane is the one settlement surface reachable over real HTTP with no funding seed — money moves onchain, so a faithful `payment_proof` (fee leg derivable from the relay's own published key) is all a delegator needs. Behavioral pair: [`booted-settlement-activation.test.ts`](../../services/relay/src/__tests__/booted-settlement-activation.test.ts) boots the compiled artifact, reads the relay key over `/federation/v1/identity`, derives the treasury address, registers a priced worker, submits a P2P task + receipt, and reads the `/agent/:id/settlements` route — accept-half (a `settlement_mode='p2p'` row with `amount_settled` = worker net and a NON-ZERO `platform_fee` equal to the proof's fee leg: the fee split preserved in the recording) and reject-half (a fee leg routed to a non-treasury address is refused at submission with `TASK_P2P_FEE_ADDRESS_MISMATCH` and records nothing: the fee-destination binding, [`settlement-authority-binding`](settlement-authority-binding.md), enforced in the artifact). Severing runs proved BOTH directions: zeroing `p2pFeeAmount` in `handleReceiptIngestion` (tasks.ts) + rebuild dist turns the accept-half red (`expected +0 to be 26316`); short-circuiting the `proof.fee_to_address !== relayTreasuryAddress` check turns the reject-half red at 201 (an over-permissive record of a fee the relay never received). The reject-half asserts the SPECIFIC error code, not a bare 400, so a bad-tx-format 400 cannot pass it for the wrong reason.
- **Honest remainder (the money-conservation heart is NOT booted-reachable):** the relay-custody lane's `settlementFunded = grossAmount === 0 || allocationClaimed` gate (tasks.ts) — the fail-closed claim that stops a resubmitted/refunded allocation from re-crediting a worker (unfunded-mint / double-credit) — is the most load-bearing property of this link, and it is deliberately NOT reachable over HTTP: funding a virtual account is a non-HTTP ledger seed since the self-declared `/deposit` route was removed as a treasury-drain vector. That severing therefore stays at the composition-root in-process tier (`createTestRelay` drives the real `registerTaskRoutes` composition) — the same "booted rung remains" shape as policy→action. Named here so the gap is visible, not silently absent.
- **Discovered by the receipt→settlement scouting, not by design (deferred-with-trigger):** the relay-custody split ignores the injected fee rate. `settleOnReceipt(allocation, receipt, null, settlementId)` at tasks.ts omits the optional 5th `feeRate` argument, so the SPLIT (`platform_fee = gross × feeRate`) falls back to the module-default `PLATFORM_FEE_RATE` (0.05) — while the GROSS LOCK one block up uses `computeGrossAmount(unitCost, platformFeeRate)` with the INJECTED rate. On any relay configured with `MOTEBIT_PLATFORM_FEE_RATE ≠ 0.05` the two disagree: the stored `platform_fee`/`platform_fee_rate` diverge from what the delegator was charged (fails safe — worker over-paid, relay under-collects — but the recorded fee is wrong). No static needle covers it (`check-money-boundary` is lexical; nothing pins that the split rate equals the lock rate). The fix is a one-line thread (`settleOnReceipt(..., platformFeeRate)`) but it is a money-path change that needs a composition-root in-process fee-rate readback to guard it, so — per the money-path caution the action→receipt wrong-fix taught — the arc records it rather than shipping it inside a test+doctrine PR. Trigger: the next relay-custody settlement change, or the first non-default `MOTEBIT_PLATFORM_FEE_RATE` deployment.
- **settlement→trust — booted rung shipped (the ladder's FINAL link).** After a settlement is recorded, the relay updates its `agent_trust` ledger — thesis #2 (the interior grows more capable over time). Two guarantees, both read back over real HTTP from the master-token `GET /api/v1/agent-trust/:id` route (the trust edge is NOT gated on the optional credential flag, so the rung is flag-independent; reputation-credential issuance stays covered in-process by `trust-flywheel-e2e.test.ts`). Behavioral pair: [`booted-trust-activation.test.ts`](../../services/relay/src/__tests__/booted-trust-activation.test.ts) boots the compiled artifact and, over the P2P lane already proven booted-reachable — accrual-half (a real cross-party P2P settlement DRIVES a persisted trust update: `interaction_count`/`successful_tasks` incremented, `trust_level` = `first_contact`) and self-dealing-half (a self-delegated settlement moves NO trust — the `!isSelfDelegation` anti-sybil guard that stops an agent minting its own reputation, the necessary core of sybil-resistance in [`agents-as-first-person-trust-graph`](agents-as-first-person-trust-graph.md)). The accrual-half is the #357-shape guarantee at the trust link, with a twist: the trust write is wrapped in a best-effort `try/catch` that SWALLOWS every error ("don't block receipt delivery"), so a dormant OR throwing trust write leaves settlement intact and every unit test green — only a booted read-back catches it. Severing runs proved BOTH directions: neutralizing both `setAgentTrust(...)` calls (level still COMPUTED by `evaluateTrustTransition`, never persisted — the #357 dormancy, here doubly hidden by the swallow) + rebuild dist turns the accrual-half red (`expected [] to have a length of 1`); dropping `!isSelfDelegation` from the guard turns the self-dealing-half red (`expected [ … ] to have a length of +0 but got 1` — a self-minted edge).
- **Discovered by the settlement→trust probe, not by design (deferred-with-trigger):** the relay-side `agent_trust` write is keyed `[motebitId (the /result URL param), receipt.motebit_id (the executor)]`. For DIRECT P2P delegation the URL is the worker, so the edge is keyed `[worker, worker]` — a self-edge that AGGREGATES across delegators: two distinct delegators A and C settling with worker B both increment ONE `[B,B]` record to `interaction_count` 2, and neither delegator holds a record (empirically confirmed with a throwaway booted probe). That is a coarse per-worker counter, NOT the first-person `[submitted_by, worker]` edge the trust doctrine describes — the doctrine's first-person, non-transitive graph lives in the AGENT RUNTIME interior, not this relay table, so the divergence is not automatically a violation. But it becomes a **global-score leak** the moment any routing path reads `agent_trust` for cross-delegator candidate ranking, which the doctrine forbids ("refusing the global score is the necessary core of sybil-resistance"). Whether the relay should instead key on `submitted_by` is a doctrine call, not a hasty rebind (the action→receipt wrong-fix taught that lesson on this exact function), so the booted rung asserts only the two guarantees unambiguously TRUE for direct P2P and does NOT assert first-person isolation on the relay. Trigger: the next change to the trust-key resolution, or the first routing path that consumes `agent_trust` cross-delegator.
- **Honest remainder (the earned-and-losable half is NOT booted-reachable):** the trust DOWNGRADE path (`downgradeP2pTrust` in `p2p-verifier.ts` — a failed/unverifiable P2P settlement must LOSE trust, never keep it) runs in a background verifier loop keyed on Solana RPC, not synchronously with the receipt POST, so it is not observable over a single booted HTTP round-trip. It stays at the composition-root in-process tier (call the verifier directly, assert `failed_tasks` moved) — and the scout found it is currently untested even in-process, a standalone coverage gap. Same "booted rung remains" shape as policy→action and the receipt→settlement `settlementFunded` gate.
- **🏁 The pipeline-link ladder is per-link COMPLETE** (identity→authz, policy→action, action→receipt, receipt→settlement, settlement→trust — every link now turns a behavioral test red when its production wiring is severed against the deployed artifact, booted where the enforcement point is HTTP-reachable and composition-root in-process where it is not, each gap named). This does NOT close the class. Seven links each severed in isolation is not the same as the whole pipeline severed in one deployed flow: the honest remainders (relay-custody `settlementFunded`, the trust downgrade, the fee-split seam, the market-candidates two-site carve-out, the embedded-key heal, the swallowing trust catch, the un-pinned `MOTEBIT_RELAY_ISSUE_CREDENTIALS` default) are real, and no single test yet drives identity→…→trust end-to-end through the booted artifact severing at each point. That whole-pipeline single-flow severing is the next escalation, not a shipped fact — consistent with the provisional discipline below.

**Discovered by the second severing, not by design:** the market-candidates carve-out is TWO sites that must agree — the catch-all's exclusion list and the route's own `dualAuth` registration — and losing only the registration leaves the route **fail-open** (fully unauthenticated), a strictly worse posture than the f9d875b1 incident's fail-closed-wrong 401. No gate covers this desync class ([`check-agent-route-auth`](../../scripts/check-admin-route-auth.ts) scopes to `/api/v1/admin/*` only), and the exclusion list mixes genuinely auth-less-by-design routes (signature-is-auth, public protocol artifacts) with routes that carry their own auth elsewhere. Deferred-with-trigger: a gate (or a single-registration seam where the exclusion entry carries its expected audience and generates both halves) lands when the next carve-out is added or the next route-auth incident occurs — whichever is first. Until then the booted authz pair behaviorally guards the one instance severed here.

The remaining honest gap: `#357`/`#358` proofs invoke their compositions in-process, and the image probe is amd64-only (the multi-arch manifest's arm64 half shares layers via the buildx cache but is never itself booted) and fires on main pushes and release tags — a publish gate, not a PR gate. All three named incidents have now turned a pass red when deliberately reintroduced — the bar this section set, `#359` at every tier from source to shipped image — but the class as a whole stays open: these are three incidents on one pipeline, not the pipeline. "We have closed the composition gap" remains an aspiration, not a fact, and this doc says so on purpose.

## Scope

Governs how the repository proves itself, alongside [`agentic-era-engineering`](agentic-era-engineering.md) (coherence is the scarce resource) and [`gate-repair-instructions`](gate-repair-instructions.md) (a gate teaches its rule on contact). It is not a wire-protocol change and introduces no new type. It is the invariant that sits _above_ the gate suite: each gate proves a part; this obligation demands proof that composition-in-deployment preserves the guarantee the parts were built to make.
