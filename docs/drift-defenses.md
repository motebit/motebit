# Drift defenses

Every architectural drift this codebase has suffered has the same shape: the canonical source of truth was invisible, unenforced, or ambiguous, so sibling copies emerged and drifted independently. The response is systematized:

1. **Name it.**
2. **Identify the canonical source of truth.**
3. **Name the sync owner, trigger, and mismatch response.**
4. **Add a defense** — CI gate, lint rule, or explicit doctrine principle in [CLAUDE.md](../CLAUDE.md).
5. **Cross-reference the defense** from any affected package or service comment.

129 invariants are enforced today. 118 run as hard CI gates via `pnpm check` (proven effective by `check-gates-effective`); one is advisory (`check-sibling-boundaries`, PR-diff scoped); ten are build-time (TypeScript `satisfies`) or test-enforced (vitest assertions) or compound rules inside an existing gate.

## Incident histories

Each defense exists because something drifted. These are the stories. They are not binding doctrine; they exist so future maintainers understand _why_ the check exists before they consider weakening it.

### The annotation-layer detour (2026-04-22 → 2026-04-24, no gate)

A product-vision drift that didn't land in a gate because its failure mode is semantic, not mechanical. Preserved as doctrine so the category of mistake is recognizable next time.

**What happened.** The `motebit-computer.md` doctrine — 321 lines, six commits of live-test-driven deepening, tagged at `motebit-computer-exploration-2026-04-21` — was retired on 2026-04-21 17:32 (commit `fdf4cd52`) with the rationale that the specific metaphor was "aesthetic framing around a product shape other agent systems ship under plainer names" and "the motebit-unique layers sit above whichever execution mode is active and don't need a bespoke scene primitive." The cross-surface contract types (`SlabItemKind`, `SlabItemPhase`, etc.) were also removed with the fair observation that no main-branch consumer had imported them.

Two separate decisions got fused into one retire commit: (a) retire the _premature cross-surface contract scaffolding_ — correct, no consumers existed; and (b) retire the _product vision_ itself — a category error, the vision had matured through six corrections and answered the question "what category does motebit occupy?" in a way nothing else did.

Over the next three days (2026-04-22 → 2026-04-24) the vision was re-derived in progressively weaker forms: `workstation-viewport.md` framed the motebit's workstation as a "shared act-space" (better than nothing but missing the first-person frame); a Phase-2 "viewport MVP" shipped a frame-dominant pane rendering the user's own screen back at them (rejected in product review as redundant recursion); an "annotation-layer" reframe corrected the specific mirror mistake without questioning the pane-as-dashboard framing; a detailed `workstation-frontend.md` spec'd an eight-state control pane in 512 lines — a better Operator with signed receipts underneath, which is category-adjacent rather than category-creating.

**The shape of the drift.** At each step the design converged to a local optimum starting from the previous step's mistake. When the frame-dominant MVP was wrong, the fix stayed at the pane layer rather than questioning the pane itself. When "the pane is a mirror of the user's screen" was named as the mistake, the fix was "don't mirror; annotate" rather than "maybe the problem is that it's a pane." Each local correction lost more of the original vision's distinctness — six embodiment modes became one; droplet physics became CSS transitions; first-person perceptual frame became second-person activity dashboard; scene-graph placement became DOM overlay.

The failure mode generalizes: **when a vision is retired, check whether the retire commit fused a tactical decision with a strategic one.** Tactical decisions (premature protocol types, ahead-of-consumers scaffolding) are often correct to undo. Strategic decisions (retiring the product vision they belonged to) require a separate justification. If both are bundled in one commit, the strategic one rides on the tactical one's momentum without being independently examined.

**What made the detour end.** A fresh-eyes principle-review on 2026-04-24: "did we contaminate the doctrine throughout the exploration, or was the exploration good?" Answer: the six-commit exploration was good. Each commit caught something from live test or review. Nothing was bolt-on. The contamination was in the retire commit's framing and in the re-derivations that compounded from it. The restoration commit (same day) brings `motebit-computer.md` back at the tagged state and retires `workstation-viewport.md` + `workstation-frontend.md`.

**What remains correct from the detour.** The Phase 1 cleanup (commit `ec508447`) that removed the URL bar + Reader iframe from the desktop/web Workstation panels is still correct — `read_url` labeled as a shared browsing surface was a category error under any doctrine. Phase 1 stays.

**No mechanical gate.** The drift pattern is "fuse tactical + strategic into one retire" — not something a linter catches. The prevention is doctrinal: when authoring a retire commit, separate "retire X because no consumer imported it" from "retire the vision X belonged to." The latter needs its own independent argument.

### 18. Routine `every` grammar ↔ `parseInterval`

Two parsers for the same string grammar is the classic drift vector: yaml validation accepts `"5x"` because the schema only checks `z.string()`, and `parseInterval` blows up at apply time with a less specific error. ### 20. motebit.yaml schema fields ↔ zod `.describe()` hover

The `motebit lsp` language server reads `.describe()` text off the live zod schema and serves it as editor hover text. ### 21. Committed `motebit-yaml-v1.json` ↔ live zod schema

`apps/cli/schema/motebit-yaml-v1.json` is motebit's published JSON Schema — the contract that third-party YAML validators (VS Code's Red Hat extension, CI actions, the dashboard) resolve via its stable `$id`. It is **generated** from `MotebitYamlObjectSchema` by `pnpm --filter motebit build-schema`, but **committed** so external tools don't need the CLI installed to validate motebit.yaml. The generation-and-commit pattern introduces the exact drift vector it is designed to exploit: the zod source gains a new field, the author forgets to rebuild, and the published contract silently misreports the shape. Defense: `yaml-json-schema.test.ts` regenerates the JSON Schema in-process and asserts structural equality with the committed file; on failure it tells the author "run `pnpm --filter motebit build-schema` and commit the result." The failure message is the fix recipe. Added 2026-04-17 alongside the public protocol publication.

### 22. Wire-format types ↔ zod schemas ↔ committed JSON Schema

Extends invariant #21 inward. `@motebit/protocol` is on the permissive floor (Apache-2.0) and type-only by invariant #10 — it cannot hold runtime validators. But third-party Python/Go/Rust implementers that want to emit or verify motebit wire artifacts (execution receipts first; discovery responses, service listings, credentials to follow) need machine-readable contracts, not TypeScript `.d.ts`. `@motebit/wire-schemas` is the Layer-1 BSL home for zod schemas that mirror every `@motebit/protocol` wire-format type plus committed JSON Schema artifacts derived from them. The three-way pin: (1) `satisfies`-style `BrandedToString<T> extends InferredReceipt` type assertions in each schema module fail `tsc --noEmit` at build time if the zod shape diverges from the `@motebit/protocol` TypeScript declaration; (2) the `drift.test.ts` roundtrip fails CI if the committed `schema/*-v1.json` drifts from the live `zodToJsonSchema(...)` output; (3) the `execution-receipt.test.ts` runtime-parse tests fail if the zod schema itself rejects receipts that real motebit emitters produce. Adding a new wire format means adding its module, adding it to the `SCHEMAS` array in `scripts/build-schemas.ts`, adding a case to `drift.test.ts` `CASES`, committing the generated JSON. Added 2026-04-17 alongside execution-receipt-v1 publication — the first machine-readable contract that lets non-motebit systems credibly participate in the protocol without bundling our runtime.

### 23. spec/\*.md wire-format types ↔ `@motebit/wire-schemas` exports

Without this gate, the `@motebit/wire-schemas` package only enforces drift on the schemas it _already_ ships — invariant #22 catches drift inside an existing schema, invariant #21 pins the committed JSON to its zod source, but neither catches the omission case. ### 32. Referent disambiguation ↔ `@motebit/semiring` disambiguation module

The fourth non-trivial semiring consumer — the one that completes the endgame map's "semiring wherever algebra is natural" line item. Hard gate, empty allowlist. Semiring-consumer tally: agent routing, memory retrieval, notability, trust propagation, disambiguation — five sites across four packages. ### 30. Trust propagation ↔ `@motebit/market` trust-propagation module

The third non-trivial semiring consumer — the one that turns "swap the semiring, change the judgment" from a promising pair into an architectural pattern. Before 2026-04-19 `packages/market/src/credential-weight.ts` performed one-pass EigenTrust-shaped aggregation over peer-issued reputation credentials, but it asked the caller for a `getIssuerTrust(issuerDid)` function and callers typically answered with only one-hop trust — the caller's direct trust in the issuer. Multi-hop propagation (Alice → Bob → Carol where Carol issues a reputation credential to Dave) required the caller to walk the credential graph itself, multiplying trust × credential-weight along chains and maxing across competing paths. That is exactly the algebra the semiring pattern collapses. Different callers would have written slightly different loops with slightly different aggregation, and admin UI would have shown a different propagated score than AI-core or the market router for the same agent — same failure mode as #28 (reputation diverged across sites). Fix: `packages/market/src/trust-propagation.ts` exposes `propagateTrust(credentials, { roots })` under `TrustSemiring` (max-times). A synthetic super-source emits weight=1 edges to each root, and one `optimalPathTrace` call per reachable agent returns both the best propagated trust and its provenance chain — so revocation of any intermediate attestation re-propagates deterministically. Defense: `check-trust-propagation-primitives.ts` with a three-condition heuristic — file references credential-graph vocabulary (≥2 of `credentialSubject` / `issuer` / `VC_TYPE_REPUTATION` / `did:key`), combines `issuerTrust` × a weight identifier inside a loop over credentials, and does not import `propagateTrust` / `buildTrustGraph` / `makeIssuerTrustResolver`. Allowlist empty at landing. Semiring consumer tally: agent routing (`@motebit/semiring`), memory retrieval + notability (`@motebit/memory-graph`), trust propagation (`@motebit/market`) — four sites across three domain packages, enough for the thesis to graduate from "it works twice" to "it's how motebit expresses judgment."

### 42. Fly-deployed service ↔ direct `better-sqlite3` declaration (native-driver parity)

The relay is the only centralized, durability-sensitive node in the architecture — `relay_receipts.receipt_json` is append-only and byte-identical per `services/relay/CLAUDE.md` rule #11; `relay_settlements` rows are the audit record behind every virtual-account debit; losing any of this breaks invariants the doctrine leans on. On 2026-04-24 a production audit found that motebit-sync had been running on **sql.js (WASM)** instead of **better-sqlite3 (native)** for ~two months, silently, since the fallback shipped in commit `3d4b4a95` (2026-02-19) as a CLI-scaffold ergonomics feature. Impact:

- `journal_mode = WAL` was a silent no-op (sql.js's pragma handler explicitly returns `undefined` for `journal_mode = …`). Production sat in `journal_mode = delete` (rollback-journal) — pre-WAL SQLite durability semantics, no concurrent-reader benefits.
- Writes flushed on a **1-second debounce** via `fs.writeFileSync` of the entire DB file (`packages/persistence/src/sqljs-driver.ts`, `FLUSH_DELAY_MS = 1000`). A crash within the debounce window lost the batched writes. Fly machines can reschedule at any time; SIGKILL is reachable.
- Write amplification was O(db_size), not O(change_size). At 5.3 MB this was tolerable; at 50 MB it would become punishing. No forcing function limits DB size.
- Litestream (the run.sh branch that streams WAL to S3) would silently replicate nothing useful if enabled, because sql.js never writes WAL. The intended DR path was broken-by-substitution.

Root cause: `@motebit/persistence` keeps `better-sqlite3` in `optionalDependencies` so the CLI scaffold (`npx create-motebit`) degrades gracefully on exotic platforms (Nix, WSL, containers without a C++ toolchain) — a legitimate design choice for that audience. But `pnpm --filter @motebit/api deploy --prod /app/deployed` builds a self-contained tree from the target service's own `package.json` dep graph, and transitive optional-of-workspace-dep is empirically dropped. `services/api/package.json` didn't list better-sqlite3 directly, so the native binding (which DID build successfully during `pnpm install --frozen-lockfile` per the deploy-action log) never made it into the runtime image. `openMotebitDatabase` at `packages/persistence/src/index.ts:2944` caught the `MODULE_NOT_FOUND` silently and substituted sql.js with no log line. No one saw the downgrade because the driver interface is faithful — correctness tests passed against both.

Defense: a fourth rule inside `check-deploy-parity.ts`. For every fly-deployed service (anything with a `fly.toml`), if the service's `package.json` depends on `@motebit/persistence`, it must ALSO declare `better-sqlite3` as a direct dependency. The rule is narrow by design — services that don't use persistence are unaffected, and the CLI scaffold's legitimate sql.js fallback path stays available. Paired with two runtime-layer fixes: (a) `services/api/src/index.ts:327` switched from `openMotebitDatabase` (fallback-friendly) to `createMotebitDatabase` (native-only, throws if missing) — any binding drop now fails loud at boot, not silently at query time; (b) the boot log adds `driver: relay.moteDb.db.driverName` to the `relay.starting` event, so every future boot prints the driver identity and drift is auditable from `flyctl logs` without SSH.

The fix is compatibility-safe: both drivers read the standard SQLite file format. The current production DB (written by sql.js) is a valid SQLite file (verified via `PRAGMA journal_mode` and `user_version` queries). On next deploy, better-sqlite3 opens the same file, the `pragma("journal_mode = WAL")` call at `packages/persistence/src/index.ts:2510` actually takes effect, and the transition happens at next write with no data loss.

Strategic prize: the relay's durability contract is now mechanically enforced. The invariant generalizes — any future service that stores state in SQLite and ships to production has one declaration point that expresses its native-driver requirement, and the gate catches the omission on PR. The sql.js fallback remains for its original consumer (scaffolded motebits on platforms without a native toolchain) without leaking into production infrastructure. Shipped 2026-04-24 alongside the tag-stripping primitive (#41) as part of the principal-engineer post-landing audit sweep.

### 41. Chat-surface tag/marker stripping ↔ `@motebit/ai-core` canonical primitive

The runtime emits three internal-narration tags (`<thinking>`, `<memory>`, `<state/>`) and two prompt-injection boundary markers (`[EXTERNAL_DATA source="…"]…[/EXTERNAL_DATA]`, `[MEMORY_DATA]…[/MEMORY_DATA]`) into the `text` stream chunks surfaces subscribe to. Every chat surface that renders those chunks must strip the set before showing them to the user — otherwise model-reasoning traces and tool-boundary markers render as chat content. Before 2026-04-24 three inline copies of the stripping pass existed across the repo: desktop's `stripPartialActionTag` in `@motebit/ai-core/core.ts` handled only `<memory>` + `<state/>` (missing `<thinking>` and both `[EXTERNAL_DATA]` and `[MEMORY_DATA]` entirely); web's `apps/web/src/ui/chat.ts` had a private `stripInternalTags` covering the full set including partial-stream fragments; and web's `apps/web/src/bootstrap.ts` had a third near-copy in `cleanConversationHistory` for history replay. Runtime chunks carrying `<thinking>` tags (every reasoning-capable provider emits these when the model plans before speaking) or `[EXTERNAL_DATA]` markers (every tool-result pass) rendered as visible chat content on desktop — a user-visible correctness bug sitting in the highest-frequency interaction surface in the repo.

Defense: `check-chat-tag-stripping.ts` scans every `.ts` / `.tsx` under `apps/*/src` (and Next.js `app/`) and `services/*/src` for `.replace(/…)` calls whose regex body contains one of seven tokens — `<thinking>`, `<memory`, `<state`, `[EXTERNAL_DATA`, `[MEMORY_DATA`, `[/EXTERNAL_DATA`, `[/MEMORY_DATA`. A file that imports `stripInternalTags` or `stripPartialActionTag` from `@motebit/ai-core` is exempt (it's composing the primitive, not shadowing it); a file that does not and contains a matching `.replace` line is drift. Tests, `dist`, `.generated.ts`, and `.d.ts` excluded. The canonical primitive itself lives in `packages/ai-core/src/core.ts` and is outside the scan.

Landing pass: extracted the full regex set into `stripInternalTags` in `@motebit/ai-core` (closed/opener/partial-closer for every token), refactored `stripPartialActionTag` to compose it plus the action-asterisk + whitespace normalization that only desktop's plain-text render path needs, wired web's `chat.ts` and `bootstrap.ts` to import the primitive, added a test battery (every tag + every marker + every partial variant + the full composed stack in one pass). The gate caught one additional inline copy on its first run (`apps/web/src/bootstrap.ts:237–239` in `cleanConversationHistory`) — the same pattern-of-patterns where the regex set gets quietly copied into every context that needs it. Post-landing: one canonical primitive, one regex set, surfaces converge. Adding a new internal tag (future `<plan>`, `<reflection>`, etc.) is one edit in `@motebit/ai-core/core.ts` plus one token in `scripts/check-chat-tag-stripping.ts`. The chat surface — the highest-frequency user interaction in the repo — can no longer drift silently. A third chat surface (mobile RN, spatial, CLI TUI) joining the repo cannot reinvent the pass without the gate firing on PR.

### 40. Canonical `@motebit/sdk` preset identifiers ↔ surface-app declarations

`@motebit/sdk` is the stable developer-contract surface (`packages/sdk/CLAUDE.md` rules 3-4). It exports the canonical preset vocabulary — `APPROVAL_PRESET_CONFIGS`, `COLOR_PRESETS`, `RISK_LABELS`, the `DEFAULT_*_CONFIG` values — as public API. A bird's-eye audit on 2026-04-24 found `apps/mobile/src/mobile-app.ts` had redefined `APPROVAL_PRESET_CONFIGS` locally with different numeric values than the SDK canonical: mobile's `balanced` preset used `denyAbove: 4` (SDK canonical: `3`), which allowed R4 Money tasks to route through the approval modal on mobile while desktop and web denied them outright; mobile's `autonomous` preset used `requireApprovalAbove: 2` (SDK canonical: `3`), which asked approval at R2 Write on mobile while desktop and web auto-allowed through R3. Same motebit identity, divergent governance posture per surface. `apps/web/src/ui/settings.ts` also redefined the constant, though its values happened to match the SDK — still a drift waiting to happen on the next SDK value change. Two adjacent identifier collisions on `RISK_LABELS` were found in `apps/desktop/src/ui/chat.ts` and `apps/mobile/src/components/ApprovalCard.tsx`; both carried surface-specific presentation shapes (`{ label, cls }` for CSS classes; `{ label, color, bg }` for RN styling) but shadowed the SDK's semantic `Record<number, string>` label table — a shape-different drift that would have made a future semantic-label change invisible to surfaces using the canonical name for the "presentation" meaning.

Defense: `check-preset-imports.ts` walks `apps/*/src` (and Next.js `app/`) for top-level declarations (`const`, `let`, `var`, `interface`, `type`, `enum`, `class`) whose name matches any entry in a closed `CANONICAL` identifier list derived from `@motebit/sdk`'s public surface. Declarations are detected by a `^(?:export\s+)?(?:const|let|var|interface|type|enum|class)\s+<name>\b` regex rooted at line starts — imports (`import { X } from`), accesses (`X.y`), string literals, and object keys do not trip it. A file that names a canonical identifier is allowed one legitimate pattern: a re-export trampoline (`export { X } from "@motebit/sdk"`) in the same file, so surface modules can preserve local import paths like `import { COLOR_PRESETS } from "./mobile-app"` while the canonical value still flows through the SDK. The re-export probe matches the identifier inside the brace list regardless of order alongside other exports. Tests are excluded by the standard `__tests__` / `.test.ts` filter — test files legitimately construct local fixtures. Exit 1 on any violation, with a message that names the shadowed identifier, the canonical source file, and the two fix options (direct import, or import + re-export trampoline).

Landing pass: mobile and web converged to import `APPROVAL_PRESET_CONFIGS` from the SDK (mobile as re-export trampoline preserving `./mobile-app` as the module path its components import from); desktop's and mobile's `RISK_LABELS` renamed to `RISK_BADGES` to stop shadowing the SDK canonical (the label-string drift — lowercase `"read"` vs the SDK's `"R0 Read"` — is a separate product UX call, flagged in the file comments, out of scope for the gate-landing rename). First clean run: 8 apps × 7 identifiers = 56 potential shadow sites, zero violations. The developer-contract vocabulary is now type-stable across every surface of the same motebit identity. A governance-preset change in `@motebit/sdk` is the single mechanism for changing behavior on every surface simultaneously; an app cannot quietly fork its own interpretation of what "Balanced" means.

Strengthened 2026-05-22 (value-shadow probe): a cross-surface drift audit found the original name-based gate had a blind spot — a copy re-encoding the preset _values_ under a **different** identifier evades the per-name regex entirely. Four such shadows had accumulated: `presetConfigs` (`apps/web/src/web-app.ts`, `apps/spatial/src/spatial-app.ts`) and `PRESET_GOV` (`apps/web/src/web-app.ts`, `apps/desktop/src/identity-manager.ts`) each re-encoded `APPROVAL_PRESET_CONFIGS` under names the gate could not see. Values happened to match canonical (no live posture divergence), but it was the same drift class the gate exists to prevent — one SDK preset change away from silently desyncing `motebit.md` governance synthesis per surface. All four now consume the canonical config by indexing it (`APPROVAL_PRESET_CONFIGS[preset]`). The gate gained a `approvalPresetValueShadowLine` probe: any literal mapping all three `ApprovalPreset` names (`cautious`/`balanced`/`autonomous`) to objects carrying governance fields (`maxRiskLevel`/`requireApprovalAbove`/`denyAbove`/`require`/`deny`) is a value-shadow regardless of variable name. The governance-field requirement keeps benign preset-keyed maps (label/icon lookups) out, so the probe targets exactly the money-safety risk. The closed `ApprovalPreset` vocabulary is what makes this detectable: a surface consumes the config by indexing it, never by re-writing its keys.

### 39. `@deprecated` annotations ↔ four-field contract + temporal sanity

`docs/doctrine/deprecation-lifecycle.md` (landed in commit `b7fa428b` on 2026-04-22) mandated a four-field contract for every `@deprecated` annotation: `since <X.Y.Z>`, `removed in <X.Y.Z>`, a replacement pointer, and a `Reason:` block. The doctrine named `check-deprecation-discipline` as the candidate gate that would enforce the contract mechanically — but the gate itself was deferred. For two days the contract was enforced by PR review only. The 2026-04-24 post-revert audit of the squash postmortem found the discipline holding (all 19 sites four-field-compliant, all 12 replacement targets reachable) but flagged the asymmetry: three state-shape migrators (`migrateAppearanceConfig`, `migrateVoiceConfig`, `migrateLegacyMobileSettings`) lacked the `@permanent` classification the doctrine's implementation pattern called for — exactly the drift a mechanical gate would catch before it reached two reviewers. Defense: `check-deprecation-discipline.ts` scans every `.ts` / `.tsx` under `packages/**`, `apps/**`, `services/**` (excluding tests, dist, coverage, `.d.ts`, `.generated.ts`), extracts the comment block surrounding every `@deprecated` token (walks both backward to the block opener AND forward to the closer, because motebit's convention places `Reason:` lines AFTER the annotation), and validates five rules: (a) `since <semver>` on the annotation line, (b) `removed in <semver>` on the annotation line, (c) `removed in` strictly greater than the containing package's current version (a past-due sunset shipped unremoved is a broken promise), (d) replacement pointer recognizable by one of five hints (`Use {@link X}` / ``Use `X` instead`` / `Pass a configured …` / `no replacement` / `rework the caller`), (e) `Reason:` line in the surrounding comment block. Baseline: 19 sites across 8 packages all passing at landing. Effectiveness probe: dropping a probe file into `packages/sdk/src/` with only `@deprecated since 1.0.0, removed in 1.1.0.` (no replacement pointer, no reason) exits non-zero with both violations named. With this landing, the three-gate deprecation pipeline is complete — this gate enforces the marker shape, `check-changeset-discipline` enforces migration guides on `major` changesets, `check-api-surface` enforces that public-API changes carry a pending changeset. The four-field contract that was convention is now invariant; drift in deprecation metadata can no longer compound silently between PR reviews.

### 38. TS `getElementById` / `querySelector('#...')` ↔ declared ids

The surface apps (`apps/desktop`, `apps/web`, `apps/inspector`) each pair a TypeScript module with a static `index.html`. The TS builds behavior by looking elements up from the HTML with `document.getElementById("...")` — a pair the TypeScript compiler can't verify. The HTML sees an attribute literal; the TS sees a string argument; the two have no link except the author's memory. When the two drift — a TS rename that misses the HTML, an HTML typo, a copy-paste from another app's markup — `getElementById` returns `null` at runtime, and whatever chained off it (`.addEventListener`, `.value = ...`) explodes mid-init. The crash kills the render bootstrap wholesale, not just the buggy feature.

The incident that forced the gate was a 37-day dormant crash. Commit `6f682fcd` on 2026-03-17 ("Key rotation UI across all surfaces") shipped desktop's rotate-key dialog. The TypeScript in `settings.ts` queried `rotate-key-reason` / `rotate-key-error` / `rotate-key-result` / `rotate-key-cancel` / `rotate-key-confirm` — five ids, all of which were written into the adjacent HTML as `rotate-reason` / `rotate-error` / `rotate-result` / `rotate-cancel-btn` / `rotate-confirm-btn`. Zero tests exercised `initSettings()` end-to-end; the compiler saw five valid string arguments. The crash sat silent for 37 days behind an earlier Buffer-polyfill drift (the Solana-ecosystem `@solana/spl-token` module referenced Node's `Buffer` at module-eval time and the desktop Vite config lacked the polyfill web had; init threw before execution reached `initSettings`). When the Buffer polyfill landed on 2026-04-23 (commit `94140605`), `initSettings()` became reachable for the first time in five weeks — at which point the null `getElementById("rotate-key-cancel")` → `.addEventListener` call took out the render bootstrap on the next boot.

Defense: `check-dom-id-references.ts` collects the set of declared ids from every surface app's `index.html` (`id="..."` or `id='...'`) and its TS sources (`.id = "..."`, `setAttribute("id", "...")`, and any `id="..."` inside `innerHTML`/`insertAdjacentHTML`/template-string HTML fragments — the third pattern catches dynamically-built chrome like confirm dialogs and toast bodies). It then finds every `document.getElementById("literal")` and `document.querySelector("#literal")` in the same TS tree and fails CI if any literal is not in the declared set. Non-literal arguments (variables, interpolated template literals) are skipped — not statically analyzable and not the drift pattern this gate catches. Tests under `__tests__` are skipped — they construct test DOM. A `scripts/check-dom-id-references.allow.json` escape hatch takes a `{ appDir: { id: "reason" } }` structure where each waiver must carry a non-empty reason string (rejected at load time, not silently accepted — waivers without a reason rot faster than the gate can catch them). First waiver: `apps/web` `max-tokens-select`, documented as an intentional residual from commit `05ba088c` on 2026-04-01 ("Remove Response Length setting from all surfaces") where the element was deleted but the TS lookup was kept with explicit `| null` typing and null-guarded use per that commit's own message ("JS handles null element gracefully"). First run against current main found one waivable case and 454 lookups that resolve cleanly.

Every id mismatch in the repo's surface layer now fails CI on day one instead of day 37. The pattern generalizes — wherever one language queries a string-identity declared by another (DOM ids, CSS class hooks, sync-channel names, IPC command names), a lookup ↔ declaration pair is a drift surface and benefits from the same static sweep.

### 51. README/CLAUDE.md `npm i -g <pkg>` / `npx <pkg>` invocations ↔ workspace package `bin` field

A new drift class that surfaced 2026-04-26 during a sibling-boundary audit on the Android Keystore + Play Integrity deprecation pass: **"code-shaped prose drift"** — documentation that looks like runnable code but no longer matches the package's actual capabilities. The triggering incident: `@motebit/verifier`'s README opened with `npm i -g @motebit/verifier && motebit-verify motebit.md` for months after the 2026-04-09 verify↔verifier swap removed the package's `bin` field. The published install snippet was silently broken — anyone landing on the npm page and copy-pasting would install the library and then fail because no `motebit-verify` binary existed in it. It survived `tsc`, `eslint`, every test runner, and every other `check-*` gate because none of those read prose. The drift was caught only by chance during an unrelated audit.

The class is broader than this one incident. Common triggers that produce the same shape: package rename / repositioning (the verify↔verifier swap), `bin` field removal (a package stops shipping a CLI), `exports` change (an API moves between packages), API rename (`oldFn` → `newFn`), major version bump with breaking changes. After any of those, README install/usage snippets need re-validation in the renamed package AND every sibling that cross-references it.

Defense: `check-readme-bin-claims.ts` walks the repo, builds an index of every workspace package's `bin` field, then greps every `README.md` and `CLAUDE.md` for `npm i -g <pkg>` / `npm install -g <pkg>` / `npm install --global <pkg>` / `npx <pkg>` patterns. For each invocation naming a `@motebit/*` / `motebit` / `create-motebit` package, the gate asserts that named package's `package.json` has a `bin` field. A package with no `bin` being named in a global-install or npx context is drift — exits non-zero with the offending doc, snippet, and target package. Out of scope: import-statement drift (`import { X } from "@motebit/foo"` where `X` no longer exists) — that requires resolving named exports against the dist surface and is more invasive than a regex sweep; if a future incident demands it, the script is the natural extension home.

The defense reuses the synchronization-invariants meta-principle one layer outward: documentation is a sibling of code, and the canonical truth (the package.json `bin` field) drifts apart from its prose copy (the README install snippet). Same shape every gate before this has guarded — name the canonical source, name the sibling copy, write the probe that asserts they agree. The lesson worth carrying: every existing motebit drift defense reads code or schemas; this is the first one that reads prose. Future drift gates that defend executable-documentation invariants (import paths, code-block examples, JSDoc `@example` tags) should follow this pattern rather than expanding the scope of an unrelated gate.

Single canonical source, zero sibling-listing drift — paid for in format brittleness. A contributor editing `DOCTRINE.md` for prose reasons (renumbering, swapping `**[…]**` for plain `[…]`, dropping the em-dash, missing a sentence-ending period) would break `pnpm build` (which runs the generator's `prebuild`) with no early warning.

Defense: `check-doctrine-format.ts` runs the same regex on every commit touching `DOCTRINE.md` and asserts four invariants — (1) every chain bullet matches the canonical format, (2) the chain length is exactly nine documents (the canonical DROPLET → CONFERENCE chain), (3) every cited filename exists at the repo root, (4) every derives-clause ends with sentence punctuation. Failure exits 1 with the specific violation; the message names the canonical format so a contributor sees exactly what to restore. Moves the failure from "next `pnpm build` somewhere downstream" to "the commit that introduced it."

Adversarial probes: (a) replace `DROPLET.md` in a chain bullet with `NONEXISTENT_PROBE.md`, confirm exit 1 with `cited file does not exist at repo root`; (b) drop the `**` bold markers from one bullet, confirm exit 1 with `parsed 8 chain bullet(s) but expected 9`. Both restored cleanly.

When the chain genuinely extends to ten (a new foundational document lands), the workflow is: append the doc to `DOCTRINE.md`, bump `EXPECTED_CHAIN_LENGTH` in this gate, add the file at the repo root. The deliberate three-way coupling is the protocol that keeps the chain coherent.

### 37. Hardware-attestation VC envelope + `attestation_receipt` parser ↔ canonical primitives

Hardware attestation is the moat layer that turns motebit's Ed25519 identity from software-custody into a hardware-rooted claim a third party can rank. The vertical is two primitives wide: the VC composer that embeds a `HardwareAttestationClaim` in a self-signed `AgentTrustCredential` envelope (`composeHardwareAttestationCredential` in `@motebit/encryption`), and the verifier that parses the claim's `attestation_receipt` JWS-shape body (`verifyHardwareAttestationClaim` in `@motebit/crypto`). Before 2026-04-22 the CLI (`apps/cli/src/subcommands/attest.ts` → `buildAttestationCredential`) and the desktop surface (`apps/desktop/src/mint-hardware-credential.ts` → `mintHardwareCredential`) each composed the envelope themselves — same `@context`, same `type: ["VerifiableCredential", "AgentTrustCredential"]` tuple, same `eddsa-jcs-2022` proof, two copies. A sibling-boundary audit consolidated them into the canonical composer; a parallel-worktree iOS / Expo mobile surface is landing as the third consumer. Without a gate, the next field added to the envelope (a future `challenge` binding, a rotated context URI, a signature-scheme flip for PQ migration) would ship on one surface and silently miss the other two. The verifier side has the same shape but the opposite asymmetry: `attestation_receipt` is an opaque JWS-style `base64url(body) . base64url(p256-signature-der)` pair, and any file parsing that pair with `.split(".")` + base64url decode + P-256 verify outside `verifyHardwareAttestationClaim` duplicates fail-closed error handling, algorithm pinning (`ecdsa-p256-sha256`), and the identity-key binding check (`body.identity_public_key` must equal the expected Ed25519 hex). Defense: `check-hardware-attestation-primitives.ts` with two rules. Rule A scans `apps/**/src`, `packages/**/src`, `services/**/src` (excluding tests and `.generated.ts` bundles) for the VC-envelope literal tuple `"VerifiableCredential", "AgentTrustCredential"` alongside a `hardware_attestation:` property assignment (type-tuple distinguishes composition from the rendering/matching sites in `@motebit/render-engine` / `@motebit/market`; excluding `hardware_attestation?:` keeps the protocol type declaration in `@motebit/protocol` legitimate); a file matching both that does not import `composeHardwareAttestationCredential` is drift. Rule B scans the same tree for any `attestation_receipt` reference combined with one of the JWS-shape parse primitives (`.split(".")` on the receipt, `fromBase64Url(`, `verifyP256EcdsaSha256(` / `p256.verify(` / the `ecdsa-p256-sha256` algorithm identifier); a file matching either without `verifyHardwareAttestationClaim` imported is drift. Canonical homes (`packages/encryption/src/hardware-attestation-credential.ts`, `packages/crypto/src/hardware-attestation.ts`, `packages/crypto/src/index.ts` — the last runs the verifier through the unified `verify()` dispatcher) are exempt by path. Allowlist empty at landing. Effectiveness probe in `check-gates-effective.ts` plants a deliberate Rule A drift (a fixture file composing the envelope without importing the canonical composer) and confirms non-zero exit. The hardware-attestation vertical is the clearest expression of "the body is passive, the interior is active" in the identity stack — a peer-verifiable hardware signal that rides inside the existing credential substrate without a cryptosuite migration, and one canonical shape per direction (compose / verify) keeps the three surfaces (CLI, desktop, mobile) producing receipts a third party can rank against a single algebra.

### 60. Client-side `/credentials/submit` callers ↔ `{accepted, rejected, errors}` body inspection

The relay's `POST /api/v1/agents/:id/credentials/submit` endpoint returns HTTP 200 even when it rejects every credential in the batch — self-issued credentials (per spec/credential-v1.md §23 "Trust cannot be manufactured"), signature-failed credentials, unknown subjects all collapse to a body-level rejection: `{accepted: 0, rejected: N, errors: [...]}`. A submitter that checks only `response.ok` (HTTP 2xx) reports success while the relay's index never accepted anything.

This is the load-bearing bug commit `63fa2199` reverted on 2026-04-25 ("revert(hardware-attestation): unwind broken publish-at-bootstrap flow on all three surfaces"). The publish helpers shipped to mobile/desktop/web checked only `resp.ok` and reported `kind: "submitted"` while the relay returned `{accepted: 0, rejected: 1, errors: ["self-issued credential rejected"]}` for every post. Every surface's tests mocked `fetch` to return `{ok: true, status: 200}`; no test exercised the relay's actual reject path. The bug shipped green, ran in production, and was caught only by a fresh-eyes review against staging that hit the live `/credentials/submit` endpoint. Memory `lesson_hardware_attestation_self_issued_dead_drop` named the mechanical detector at the time — "response.ok check on /credentials/submit without inspecting accepted/rejected body counts" — but the detector never landed as code.

The post-mortem on 2026-04-28 found the same anti-pattern reincarnated in `packages/runtime/src/interactive-delegation.ts:84-89`. The peer-flow Phase 1 ship that came after the revert wired credential submission through the runtime's `setCredentialSubmitter`, and the new submitter — called for _legitimately_ peer-issued credentials (issuer ≠ subject), so most posts succeed — checked only `resp.ok`. A malformed credential, a signature failure, or any future server-side filter would slip silently. Same bug shape, different location, three days later.

Defense: `check-credentials-submit-response-shape.ts` walks `packages/**`, `apps/**`, `services/**` TypeScript (excluding tests, `.generated.ts`, the relay's own `services/relay/src/credentials.ts` server-side handler that _emits_ the shape rather than consuming it) and finds every real `fetch(...)` call against a URL containing `/credentials/submit` — multi-line fetch with the URL on a continuation line is caught via a 500-char lookahead from each `fetch(` opening, ruling out comment-only mentions of the path that share a file with unrelated `fetch` calls. Any file with a real submitter must reference both `accepted` and `rejected` as identifier tokens — the cheapest evidence the response body is being parsed and inspected. False positives (file mentions both tokens for any reason) are accepted as cheap; false negatives (a real submitter that bypasses body inspection) would re-open the original incident class.

The fix in `interactive-delegation.ts` parses the body, reads `body.rejected ?? 0`, and logs a `body_rejected` warning with `{accepted, rejected, errors, targetMotebitId}` when the count is non-zero. HTTP failures still log `http_failed` separately so the two failure modes are distinguishable in the runtime log. Effectiveness probe in `check-gates-effective.ts` plants a fixture submitter that posts to `/credentials/submit` and checks only `resp.ok`; the gate fires with the missing-token message on both `accepted` and `rejected`. First clean run after the fix: 1 submitter, conformant.

The doctrinal generalization: HTTP 200 ≠ business success when the response body carries an `accepted`/`rejected` count. Status codes describe the transport; body shapes describe the outcome. Any endpoint that filters at the body level — credential submission, batch withdrawal, federation peer add, settlement submission — is in the same class. This gate covers `/credentials/submit` because that's where the incident surfaced; if a second body-rejecting endpoint emerges, the next gate clones this one with a different URL token.

### 81. Prompt density baseline ↔ five-question audit at growth

The system prompt accumulates rule-shaped clauses by the same mechanism every drift class has: each witnessed bug produces a clause, each clause is individually justifiable, and there is no forcing function that requires a doctrine moment at addition time. Stack 50 of them and the prompt is a configuration file disguised as a teaching — the AI is a rule-follower rather than an emergent agent, and the §4 thesis from `THE_EMERGENT_INTERIOR.md` (architecture-shapes-emergence; do not pressure the system prompt) starts contaminating across an arbitrary content domain rather than just the original `narrateEconomicConsequences` case.

The 2026-05-12 session that produced `docs/doctrine/runtime-invariants-over-prompt-rules.md` ran a prompt-prune pass that took the system prompt from a peak of ~10 clauses-added-this-session down to a local minimum of 62 rule-shaped lines. Without a gate, the next perception bug, the next observed failure mode, the next reviewer suggestion would have started the climb again — the user's subconscious surfacing the emergent-interior memo mid-session is the diagnostic signal that the climb was already underway.

Defense: `check-prompt-density.ts` counts lines in `packages/ai-core/src/prompt.ts` matching `^- ` (markdown-style bullets inside the template literals) or `^[0-9]+\. ` (numbered RULES, e.g. `INJECTION_DEFENSE`). The union is the count; today's measurement is `BASELINE = 62` inlined in the script. Growth fails the gate with the doctrine's five-question audit reproduced inline so the failure message itself is the prompt to ask the questions. Pruning is permitted silently — the gate only fails on growth.

Smoke-alarm shape, not a per-clause registry: false negatives on inline non-bullet rules ("Never narrate physical actions" embedded in a paragraph, "Don't bridge to 'what's typical'" inside a longer bullet) are accepted as the cost of low maintenance. The most common drift pattern — a new bullet added to fix a witnessed bug — is what the gate is calibrated for. Bumping the baseline is the moment of doctrine consideration: the commit that bumps it MUST name the new clause's grade (A = runtime-backed, B = teaching with named justification per the doctrine's recent-fixes table) in the message. The gate cannot enforce that the commit message contains the grading — that's the human discipline the gate forces a moment for, not a thing the gate can mechanize.

Effectiveness probe: appending a single `- Synthetic new rule` line to the prompt produces `count=63, baseline=62, +1` and exit 1 with the five-question audit printed verbatim. Restored after probe; clean run confirms 62/62. Doctrine cross-cut to `docs/doctrine/typed-truth-perception.md`: typed-truth-perception fields are the canonical A-grade alternative — when a new clause's failure pattern can be expressed as a wire field + dispatch enforcement + prompt teaching triple, the wire half goes through `check-typed-truth-perception` (#80) and the prompt half compresses to a thin teaching that this gate happily counts as one bullet. The two gates compose: #80 guards the wire half against drift; #81 guards the prompt half against accumulation.

### 82. Universal slash commands ↔ every chat-surface's slash registry

The trust-accumulation visibility arc and the onboarding arc each landed a canonical slash command in `@motebit/runtime`'s shared command dispatcher across 2026-05-12: `cmdTrust` (the 5-dimension projection — memories + conversations + signed receipts + signed deletions + federation peers) and `cmdWelcome` (the onboarding tour naming the three thesis pillars). Both are "universal commands" — they surface canonical thesis-pillar projections, live in the shared dispatcher, and have the same meaning on every surface. The contract is "every chat surface MUST register every universal command"; without a gate, the contract is silent and a new chat surface (operator console, headless agent surface, future federation peer dashboard) could ship without a universal command and the thesis claim that accumulated trust + onboarding are visible on every surface would degrade quietly at user encounter rather than at CI.

This gate originally landed as `check-trust-slash-cross-surface` locking only `/trust`. When `/welcome` shipped hours later, the obvious-but-wrong move would have been to add a sibling `check-welcome-slash-cross-surface` with the same shape and a different command name — mechanical duplication. The right move was to generalize: the gate now iterates a UNIVERSAL_COMMANDS × SURFACES matrix. Adding the third universal command costs one registry line, not a new script.

Defense: `check-universal-slash-coverage.ts` iterates the closed `UNIVERSAL_COMMANDS` registry (today: `trust`, `welcome`) against the closed `SURFACES` registry (today: web, desktop, mobile, CLI). Each surface has a `patternFor(cmd)` builder that constructs a surface-native registration regex from the command name: web/desktop/mobile build `\\bname:\\s*"<cmd>"`; CLI builds `\\busage:\\s*"/<cmd>(?:\\s|")` (the trailing optional-space-or-quote anchor keeps `/cmd` and `/cmd <arg>` matching but rejects `/cmdfoo`). The patterns are conservative on purpose — a surface that grows a different shape (e.g. desktop adopting `command: "<X>"`) adjusts at one site, not N.

Spatial is intentionally out of scope. The AR-glasses prototype has no chat surface, no slash menu; its calm-AR primitives (creature / satellite / environment / attractor / presentation) don't compose with a slash registry. If spatial gains a chat surface later, the gate's `SURFACES` list grows. Same out-of-scope discipline as `check-docs-slash-claims` explicitly excluding desktop/mobile slash menus from CLI-claim validation — different registries, separate concerns.

Effectiveness probe: renaming `usage: "/welcome"` to `usage: "/wlcm"` on the CLI surface produces `1 universal command(s) missing surface registration` with the per-command + per-surface breakdown reproduced verbatim and the canonical-shape recovery message printed. Restored; clean run confirms `2 command(s) × 4 surface(s) — all cells covered`. Doctrine cross-cut to `docs/doctrine/runtime-invariants-over-prompt-rules.md` § trust-accumulation visibility arc + onboarding arc.

### 106. Property-based test floor ↔ safety-critical packages

The motebit thesis claims architectural integrity at four boundaries that hand-written tests structurally cannot exhaust: money-path conservation (`@motebit/virtual-accounts` — `Σcredits − Σdebits − Σfees = Σbalances` across arbitrary operation sequences), hardware-attestation cryptographic invariants (`@motebit/crypto-{appattest,android-keystore,webauthn,tpm}` — any single mutation in a verified receipt MUST yield `valid: false`), parser typed-error envelopes (`@motebit/skills` — `parseSkillFile` returning `SkillManifest` OR throwing `SkillParseError` across arbitrary input bytes, including Unicode and YAML-shaped adversarial content), and semiring algebra laws (`@motebit/protocol` — associativity, identity, distributivity across arbitrary weighted-digraph inputs).

The behavioral-coverage arc landing 2026-05-20 shipped property suites across the package floor: virtual-accounts conservation (commit `08aea3db`, eleven properties), skills parse fuzz (commit `ce31fb8e`, eight properties), and per-package mutation suites for all four hardware-attestation siblings (commits `4ac42c56`, `8863c6e8`, `36fb63f4`, `00549c60`). The 2026-05-21 follow-on added the sensitivity-ladder join-semilattice laws to `@motebit/protocol`, the proof-composability laws to `@motebit/crypto` (`signBySuite`/`verifyBySuite`/`canonicalJson` — round-trip soundness, any-byte message + signature mutation rejection, wrong-key rejection, JCS key-order independence, end-to-end sign-over-canonical-body; the root every signed artifact in the system depends on, previously covered only by hand-picked example artifacts), and ported the money-path conservation suite to `services/relay`'s production `SqliteAccountStore`. That port was not a free transfer: running the same invariants against SQLite surfaced a real ordering-stability bug — `getTransactions` ordered by `created_at DESC` only, so two transactions in the same millisecond returned in arbitrary order, breaking the per-row `balance_after` monotonicity the history view depends on. The fix added `rowid` (SQLite's monotonic insertion counter) as a deterministic tiebreaker. The in-memory virtual-accounts suite structurally could not reach this bug because the in-memory store returns transactions in insertion order; only the SQLite-backed store has the tie ambiguity. This is the canonical demonstration of why production-code-path property coverage is a distinct floor entry, not a transitive consequence of package-level coverage. Without a gate, a future contributor could delete a property suite, drop the `fast-check` devDependency, or extract a new safety-critical surface without carrying the discipline forward — the eight-surface coverage would degrade silently at the next refactor.

Defense: `check-property-test-floor.ts` iterates `PACKAGES_REQUIRING_PROPERTY_TESTS` (the closed list of seven safety-critical packages, each with a reason string naming the invariant the property suite covers). For each entry it asserts TWO conditions: (a) the package's `package.json` `devDependencies` declares `fast-check`, and (b) at least one `.test.ts` under `src/__tests__/` carries a `from "fast-check"` import. Pure structural check; both conditions are simple file reads. Same closed-list-with-reason shape as `check-skills-cross-surface` (#73) and `check-typed-truth-perception` (#80) — adding a new safety-critical package costs one registry line, not a new script.

Per `feedback_legibility_ratio`: every architectural commitment ships in three artifacts — code (the eight property suites), doctrine (`docs/doctrine/evals-as-attestations.md` § "What ships now"), and a drift gate (this one). The first two existed; the third closes the loop. Demoting a package from the floor is intentional doctrine work — the commit removing an entry from `PACKAGES_REQUIRING_PROPERTY_TESTS` MUST name WHY the property promise no longer holds for that surface (the package is no longer published, the invariant moved elsewhere, the surface was subsumed by a sibling). The gate cannot enforce that the commit message contains the reasoning — that's the human discipline the gate forces a moment for, not a thing the gate can mechanize.

Effectiveness probe deferred: the `check-gates-effective.ts` meta-probe family runs every gate against a deliberate perturbation (deleting an import, dropping a devDep) as a separate CI job. The probe for this gate would temporarily remove `fast-check` from one package's `devDependencies` and assert the gate fires; landing the probe alongside this gate is the natural follow-up. Sibling cross-cut to #80 (typed-truth-perception) — both gates lock multi-artifact commitments by structural enforcement at the inventory boundary.

## How to add a new defense

When a new drift pattern is observed:

1. **Don't just fix the instance.** Fixing `foo.ts` without a probe means the sibling files will drift the same way next week.
2. **Write the probe before the fix.** Running the probe against the broken state confirms it catches what you intend to catch. Adversarial test: plant a deliberate drift in a file, run the probe, confirm it exits non-zero with a specific message. Then fix and confirm green.
3. **Register in three places:**
   - `package.json` scripts (`"check-foo": "npx tsx scripts/check-foo.ts"`)
   - `scripts/check.ts` `GATES` array (with a `defends:` one-liner naming the invariant, not the mechanism)
   - This file — append a row to the inventory and a paragraph to the incident histories
4. **Link from affected code.** If the probe exists because of a specific file or package, a one-line comment there pointing back (`// see scripts/check-foo.ts`) means the next reader understands the constraint before they trip it.

The probe is cheaper than the incident. Every gate we have exists because an incident was cheaper to enforce than to repeat.
