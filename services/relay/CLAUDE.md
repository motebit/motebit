# services/relay — the relay

The only function with a legitimate centralization premium (multi-device sync). Also: settlement, federation, discovery, rate limiting, disputes, credential anchoring.

Modules: `index`, `federation`, `task-routing`, `credentials`, `pairing`, `data-sync`, `accounts`, `a2a-bridge`, `logger`, `discovery`, `migration`, `disputes`, `p2p-verifier`, `transparency`, `settlement-rails/*`, `webhooks/*`.

## Rules

1. **Never inline protocol plumbing.** Audit the package layer first: `@motebit/protocol` → `@motebit/crypto` → `@motebit/encryption` → `@motebit/mcp-client` → `@motebit/mcp-server` → `@motebit/runtime` → `@motebit/core-identity` → `@motebit/identity-file`. If none match, a primitive is missing — add it to the right package, consume here. `check-service-primitives` catches inline `fetch` to motebit endpoints, JSON-RPC strings, direct `signExecutionReceipt` calls, and `canonicalJson`/`sha256` constructing protocol-shaped payloads.
2. **`SettlementRailRegistry.register()` accepts only `GuestRail`.** The registry and the three concrete guest rails (`StripeSettlementRail` fiat, `X402SettlementRail` protocol, `BridgeSettlementRail` orchestration) live in `@motebit/settlement-rails` (Layer 1); the `@ts-expect-error` negative proof is in that package's `__tests__/custody-boundary.test.ts`. The `/health/ready` rail manifest advertises only guest rails. Sovereign rails are agent-custody and live in `@motebit/wallet-solana`. Rails accept an injected `logger` — the relay supplies its module-scoped `createLogger` adapter at construction.
3. **Structured logging.** `createLogger(module)` emits JSON with `x-correlation-id` propagated through request lifecycles. Never `console.log` in production paths.
4. **Rate limiting is per-IP, per-connection, per-peer.** 5-tier fixed window (auth 30/min, read 60/min, write 30/min, public 20/min, expensive 10/min). WebSocket 100 msg/10s per connection. Federation 30 req/min per peer. Task queue hard-capped at 100K. Per-submitter queue limit 1000/agent (HTTP 429).
5. **Fail-closed on auth.** Signed bearer tokens require `aud` audience binding. The closed registry of canonical audiences lives in `@motebit/protocol`'s `TokenAudience` literal union (see `packages/protocol/src/audience.ts`); `check-audience-canonical` enforces every signing site against `ALL_TOKEN_AUDIENCES`. Today's set: `sync`, `device:auth`, `pair`, `rotate-key`, `push:register`, `task:submit`, `admin:query`, `proposal`, `account:{balance,deposit,withdraw,withdrawals,checkout}`, `browser-sandbox-grant`, `browser-sandbox`. Cross-endpoint replay defense: a token bound to one audience is rejected by an endpoint expecting another (each route configures its own `expectedAudience` via `dualAuth`/`verifySignedTokenForDevice`); a signing-side typo would have been a runtime 401 pre-registry, now a compile error. `InboundCredentialVerifier` adapter takes precedence over static `authToken`.
6. **Relay is a convenience layer, not a trust root.** Every truth the relay asserts (credential anchor proofs, revocation memos, settlement receipts) is independently verifiable onchain without relay contact. If you add a relay-only assertion, justify why it is not protocol-shaped.
7. **Federation circuit breaker.** Per-peer forward tracking; automatic suspension at 50% failure rate over ≥6 samples. Heartbeat handles liveness (3 missed → suspend, 5 → remove). The three-state engine lives in `@motebit/circuit-breaker` (Layer 1) — `services/relay` instantiates it in `task-routing.ts` and injects a module-scoped logger. Adding a second caller (MCP client retry, webhook delivery) means a second `new CircuitBreaker({ logger })` at the consumer, not a wrapper here.
8. **P2P settlement path** is policy-based (`evaluateSettlementEligibility` in `task-routing.ts`): both parties opt in via `settlement_modes`, worker declares `settlement_address`, trust ≥ 0.6, interaction count ≥ 5, no active disputes between pair. **Identity binding is an additive eligibility signal, never a gate:** a sovereign-bound worker (offline `verifySovereignBinding` over the registry `public_key`) qualifies at a reduced cold-start bar (`first_contact` + 2 interactions) because its `motebit_id` commits to its key and so cannot be a throwaway sybil. The branch sits _after_ the strict branch, so unbound/legacy workers are unaffected; it never fabricates trust. Same shape as the additive `HardwareAttestationSemiring` (rule on hardware attestation) — see [`docs/doctrine/identity-binding-verification.md`](../../docs/doctrine/identity-binding-verification.md) § "Binding strength feeds settlement". P2p tasks skip virtual-account allocation; money moves onchain in a **single atomic Solana multi-output transaction** composing two SPL Transfer instructions — delegator→worker (the worker's listing unit_cost, recorded as `amount_settled`) and delegator→relay_treasury (the 5% platform fee, recorded as `platform_fee`). Treasury address is the relay's identity-derived Solana wallet (`deriveSolanaAddress(relayIdentity.publicKey)` — same address used by `OperatorSolanaTransfer` for Path 0 and `SolanaMemoSubmitter` for anchoring). Relay records audit with `settlement_mode='p2p'`. Async `startP2pVerifierLoop` in `p2p-verifier.ts` polls Solana RPC, walks `transfers[]` for both legs, transitions state to `verified` only when both legs match the declared amounts + addresses (mismatch → `failed` + trust downgrade). Arc 2 of the off-ramp arc landed this fee-on-P2P composition; the sibling-doc contradiction the settlement_mode arc surfaced (top-level "5% applies through both lanes" vs this rule's pre-Arc-2 "Fee: zero on P2P") is now closed. Pre-Arc-2 zero-fee P2P rows remain verifiable via the verifier's backward-compat branch (skips fee-leg check when `platform_fee === 0`). Doctrine: [`docs/doctrine/off-ramp-as-user-action.md`](../../docs/doctrine/off-ramp-as-user-action.md) § "What Arc 1 did NOT close" + the Arc 2 close. **Arc 3.5 (in flight on `arc-3.5-p2p-proof-gate`):** the `TASK_P2P_PROOF_REQUIRED` submission gate that rejects deposit-funded paid direct delegation is **staged as a deferred predicate comment at `tasks.ts:~1769`** (validated: with it active the migrated tests pass, un-migrated relay-custody delegations 402). The final arc commit re-applies it as code — do NOT let a refactor strip that comment block. Multi-hop sub-submissions are gated like direct delegation (not a carve-out); only the `settleSubReceipt` relay-write is a deferred residual. See [`docs/doctrine/off-ramp-as-user-action.md`](../../docs/doctrine/off-ramp-as-user-action.md) § "Arc 3.5 gate scope".
9. **Withdrawal hold during dispute window** — `AccountStore.getUnwithdrawableHold` (Layer-1 `@motebit/virtual-accounts`, implemented by `account-store-sqlite.ts`) sums recent relay settlements < 24h not yet disputed. `requestWithdrawal` pre-checks available balance before atomic debit. P2p settlements explicitly excluded. `accounts.ts` is a deprecation shim that routes the legacy functional API through a per-db `SqliteAccountStore`; new code consumes `@motebit/virtual-accounts` directly.
10. **Operator transparency declaration is the source of truth** (`transparency.ts` + `PRIVACY.md`). The signed JSON at `/.well-known/motebit-transparency.json` and the committed `PRIVACY.md` both derive from `DECLARATION_CONTENT`. Any code change that adds a processor, a retained field, or a retention window must update `DECLARATION_CONTENT` in the same PR — the sibling-boundary test in `__tests__/transparency.test.ts` enforces sync, but a code-vs-declaration drift (e.g., adding Sentry without naming it) the test cannot catch. Manual review for the doctrine in `docs/doctrine/operator-transparency.md`.
11. **`relay_receipts.receipt_json` is append-only and byte-identical.** The column must equal `canonicalJson(receipt)` at write time; any mutation breaks the auditor's signature reverification path (the whole point of the archive — `spec/execution-ledger-v1.md` §11.1 Storage). Persist through `persistReceiptChain` in `receipts-store.ts`; read through `getStoredReceiptJson` and return the bytes verbatim. Never `JSON.parse` + re-serialize through a non-JCS path on the way out.
12. **Aggregated withdrawals debit at enqueue time.** `enqueuePendingWithdrawal` in `batch-withdrawals.ts` debits the virtual account before inserting the `relay_pending_withdrawals` row; the fire path (serial or `withdrawBatch`) does NOT re-debit. A rail failure parks the row as `failed` with the debit still in place — the debit is the audit trail that funds were claimed. Never short-circuit the debit-before-insert order; doing so opens a double-spend window against concurrent sweeps. See `spec/settlement-v1.md` §11.2.
13. **Native SQLite driver is mandatory — no silent sql.js fallback.** The relay is the only centralized, durability-sensitive node; it MUST run on `better-sqlite3` (native, WAL-enabled, per-commit fsync) and NEVER on `sql.js` (WASM, WAL no-op, 1-second-debounced full-file rewrites). `services/relay/package.json` declares `better-sqlite3` as a direct dependency so `pnpm deploy --prod` flattens it into the runtime image — without this, the binding is dropped (it's an `optionalDependency` of `@motebit/persistence` for the CLI scaffold's exotic-platform fallback) and the service silently degrades. Boot path uses strict `createMotebitDatabase` (throws if missing), not fallback-friendly `openMotebitDatabase`. The `relay.starting` log includes `driver` so every boot prints the driver identity. Enforced by `check-deploy-parity` rule 4 (drift-defenses #42); historical incident in `docs/drift-defenses.md` §42.

14. **External medium plumbing speaks motebit vocabulary, not provider vocabulary.** Provider-native types — webhook schemas (`Stripe.Event`, `Bridge.*`), SDK calls, chain JSON-RPC envelopes (`eth_getLogs`, `eth_blockNumber`) — never leak into the relay's business logic. Each external boundary ships an adapter that (a) owns the wire format and signature/error primitives and (b) maps to a closed motebit-shaped type before handing to the consumer. Webhook adapters live under `webhooks/*`, return `null` on signature failure, and surface unknown events as `{ kind: "ignored", type }` so the switch stays exhaustive. Chain-RPC adapters live next to their consumer (e.g., `deposit-detector/rpc-adapter.ts`) and collapse every wire-level failure (network / non-2xx / JSON-RPC error / malformed body) to a single thrown `Error` — the state-machine's try/catch already handles that shape. Concrete today: `StripeSubscriptionEventAdapter` in `webhooks/stripe-webhook-adapter.ts` (consumed by `registerProxyTokenRoutes`); `HttpJsonRpcEvmAdapter` in `deposit-detector/rpc-adapter.ts` (consumed by the deposit-detector state machine). The medium-plumbing extraction mirrors the guest-rail pattern in rule 2: external APIs churn on their own cadence and must not couple to account-crediting logic.

15. **x402 facilitator construction goes through `x402-facilitator.ts` — never inline `new HTTPFacilitatorClient(...)`.** The single canonical adapter chooses CDP-authed mainnet vs. unauthenticated testnet based on env shape, and **fails fast with `X402ConfigError` when `X402_TESTNET=false` is set without `CDP_API_KEY_ID` + `CDP_API_KEY_SECRET`**. The default `https://x402.org/facilitator` is testnet-only and silently passes startup health checks while rejecting mainnet networks at route-registration time — a half-mainnet trap the centralized constructor closes. Two consumers today: `index.ts` (settlement-rail registration) and `tasks.ts` (x402 HTTP middleware). Adding a third consumer means a third call to `createX402FacilitatorClient`, never a fresh inline construction. Mainnet credentials (CDP) are JWT-bearer signed per-request via `@coinbase/x402`'s exported `facilitator` config; testnet uses the structural `{ url }` shape with no auth.

16. **Treasury reconciliation is a sibling-but-distinct primitive to the deposit-detector, with EVM and Solana sides both wired.** Both reconcilers (EVM and Solana) are "internal-ledger ↔ onchain" reconciliation pipes; the canonical doctrine on why they must NOT be unified with the deposit-detector — privilege-escalation surface (any in-database `agent_id` is potentially exposed to client-API paths) + circular-fee surface (treasury must never appear as a settlement counterparty) — lives in [`packages/treasury-reconciliation/CLAUDE.md`](../../packages/treasury-reconciliation/CLAUDE.md) Rule 1. Don't duplicate the reasoning here. **EVM side:** algebra in `@motebit/treasury-reconciliation`, relay-side wiring in `services/relay/src/treasury-reconciliation.ts`; loop runs only when `X402_TESTNET=false` AND `X402_PAY_TO_ADDRESS` is set AND the chain has a registered USDC contract + RPC URL (otherwise emits `treasury-reconciliation.disabled` log line); audits x402 facilitator settlements. **Solana side:** algebra in `@motebit/wallet-solana` (`OperatorSolanaTreasuryReconciler`, sibling of `OperatorSolanaTransfer`), relay-side wiring in `services/relay/src/solana-treasury-reconciliation.ts`; loop runs whenever `SOLANA_RPC_URL` is set (alongside the P2P verifier; pairing enforced by drift gate #104 `check-solana-treasury-reconciliation`); audits Arc 2 P2P fee-leg accumulation in the relay's identity-derived Solana treasury wallet. Both write to the same `relay_treasury_reconciliations` table, discriminated by canonical CAIP-2 `chain` column (per CAIP-30 — `eip155:8453` for EVM-Base, `solana:5eykt4UsFv8P8NJdTREpY1vzqKqZKvdp` for Solana mainnet; the same identifiers the credential-anchor + consolidation-receipt schemas use, so cross-table joins by chain compose). Admin surface: `GET /api/v1/admin/treasury-reconciliation` returns `chains[]` (one entry per chain) + cross-chain `records[]` audit feed (master-token gated per drift gate #61). Conservative phase-1 invariant: `consistent = onchain >= recordedFeeSum`; positive drift covers direct deposits + manual operator sweeps (which produce false-positive negative drift until phase 2's per-movement ledger lands — known limitation, documented in [`docs/doctrine/treasury-custody.md`](../../docs/doctrine/treasury-custody.md) § Phase 1 step 6 + § "Solana p2p-fee reconciliation"). Cite `feedback_settlement_architecture` memory.

17. **Every `app.get(...)` in `state-export.ts` routes through `emitSignedExport(c, "<artifact-type>", body)`.** The closure-scope helper JCS-canonicalizes the response body, signs it via `signContentArtifact` with `relayIdentity`, and emits the outer `ContentArtifactManifest` in the `X-Motebit-Content-Manifest` HTTP header. Witness-composition: the relay attests "this is what I assembled at time T," signed by the same identity that signs `/.well-known/motebit-transparency.json` — so a third party who pins the relay's key from the transparency declaration can verify every state-export bundle offline (`motebit-verify content-artifact <body> --producer-key <hex>`). The 12 closed-registry `ContentArtifactType` values live in `@motebit/protocol/src/artifact-type.ts` (drift-locked by `check-artifact-type-canonical` #85); the wiring requirement is drift-locked by `check-state-export-signed` (#86) — every new `app.get(...)` in `state-export.ts` MUST consume the helper or fail CI. Layered model on `execution-ledger`: the spec-1.0-compliant inner body keeps its agent-signature omission (spec §6 — relay does not hold the agent's private key); the outer envelope attests only to bundle assembly. Doctrine: [`docs/doctrine/nist-alignment.md`](../../docs/doctrine/nist-alignment.md) §8 + [`docs/doctrine/self-attesting-system.md`](../../docs/doctrine/self-attesting-system.md). The `app.delete(...)` (memory tombstone) is a mutation, not an export — out of scope for the gate; would need its own signing decision if added.
